# 修正・変更フロー

<!--
このスキルの役割:
「修正指示→計画→設計書→テスト仕様書→テスト→実装→検証」という
仕様先行・テスト先行の流れを固定し、docs・テスト仕様書・テスト・実装の
4者が常に同期した状態を保つ。
-->

## 全体の流れ

```
修正指示
  → 1. 実装計画(規模判定を含む)
  → 2. 設計書修正(仕様が変わる場合のみ)
  → 3. テスト仕様書修正 = 確認観点の合意
  → 4. テスト実装
  → 5. red 確認(新テストが落ちることを確認)
  → 6. 実装
  → 7. 検証(verify + レビュー + 画面があれば ui-check)
  → 8. 報告
(不整合が出たら: テスト仕様書を正として、テストか実装を修正)
```

## 1. 実装計画と規模判定

修正内容を確認し、まず次の3分類のどれかを判定して計画に明記する。
不明点・矛盾があれば推測で進めず、実装前に質問する。

| 分類 | 判定基準 | 適用工程 |
|---|---|---|
| 仕様変更 | 画面項目・バリデーション・権限ルール・データ構造など docs/ に書かれた仕様が変わる | フルフロー(2〜8 すべて) |
| バグ修正 | 仕様は正しく、実装が仕様どおりでない | 2 をスキップ。3〜4 は「再現する回帰テスト(`回帰:` プレフィックス)を先に書く」に読み替える |
| 軽微修正 | リファクタ・文言・スタイル等、仕様も確認観点も変わらない | 2〜5 をスキップ。テスト仕様書は観点が変わった場合のみ事後同期 |

## 2. 設計書修正(仕様変更のみ)

- `docs/screens.md`・`docs/schema.md` 等の該当箇所を先に修正する
  (docs は「今の正解」— 実装より先に正解を更新する)
- 改訂経緯・日付は書かない(最終仕様+理由のみ)
- DB スキーマが変わる場合は db-migration スキルの手順に従う

## 3. テスト仕様書修正(確認観点の合意)

- `docs/tests/` の該当カテゴリファイルで、対象関数の観点表を修正・追加する
  (カテゴリ分類と記載粒度は `docs/tests/README.md` 参照)
- **この時点でユーザーに観点を提示して合意を得る**。実装後の手戻りを防ぐのが目的
- バグ修正の場合は、再現ケースを `回帰:` プレフィックス付きの観点として追加する

## 4〜5. テスト実装と red 確認

- 合意した観点表どおりに `lib/*.test.ts` を実装する(it 名 = 観点の文言)
- **実装前に `npx vitest run <対象ファイル>` を実行し、新テストが落ちることを確認する(red)**。
  実装前に通ってしまうテストは何も検証していない — この時点で観点かテストの誤りを疑う
- テスト対象のロジックが Server Component / Server Actions に埋まっている場合は、
  `lib/` の純関数に切り出してからテストする(このプロジェクトのテスト方針)

## 6. 実装

- 実装の型(Server Component 基本、Prisma シングルトン経由、認可チェック等)は
  CLAUDE.md と new-screen スキルの規約に従う

## 7. 検証

1. `npx vitest run <対象ファイル>` で新テストが通ることを確認(green)
2. `npm run verify` を通す
3. code-reviewer サブエージェントでレビューし、must 指摘を解消する
4. 画面に関わる修正なら ui-check(Playwright)で実挙動を確認し、
   `npm run dev` を起動したままにしてユーザーが手動確認できる状態で報告する
   (テストは lib 層の単体テストのみ — 画面の実挙動は ui-check が補完する)

## 不整合時の判断基準(正の優先順位)

テストと実装の結果が食い違ったときは、次の順で「正」を判断する:

1. **正はテスト仕様書(=合意済みの観点)**。テストコードが仕様書と食い違っていればテストを修正する
2. テスト仕様書・設計書そのものが怪しい場合は、実装に合わせて書き換えず、
   **どちらが正か人間に確認する**(CLAUDE.md の方針どおり)

## チェックリスト(完了報告の前に自己確認)

- [ ] 規模判定(仕様変更/バグ修正/軽微修正)を計画時に明示した
- [ ] 仕様変更なら docs/ を実装より先に修正した
- [ ] テスト仕様書の観点についてユーザーの合意を得た(仕様変更・バグ修正)
- [ ] 新テストの red → green を両方確認した
- [ ] docs/tests/ の観点表・ケース数が実装後のテストと一致している
      (`docs/tests/README.md` の索引のケース数も含む)
- [ ] verify が通り、must レビュー指摘を解消した
- [ ] コミットはユーザーの指示を待つ

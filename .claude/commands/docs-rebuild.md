# docs / plans 再整備

## 背景と目的
度重なる改修で実装が大きく変わり、docs が追従できていない(一部のみ最新化済み)。
現在の実装を正として docs / plans を初版のように全面的に書き直す。
旧docs・旧plansは新docs・新plansを作成する際の参考として読み込んだ後、
ワーキングツリーから完全に削除する(git履歴には残る)。
新docs・新plansはレビュー対象兼、将来の再構築リファレンスとして使う。

## 最終的なディレクトリ構造
docs/
├── plans/            # 新plansのみ(番号順=理想的な実装順序)
│   ├── 000-setup.md
│   ├── 001-auth.md
│   └── ...
└── (新docs各ファイル)
※旧docs・旧plansのファイルは残さない。

## Step 0: 事前確認
- scripts/verify.sh とテスト一式を実行し、全て通ることを確認する。
  失敗がある場合は私に報告し、先に修正するか docs 整備を進めるかの指示を仰ぐこと。

## Step 1: 新docsの作成
- 情報源は現在の実装を正とする(schema.prisma、app/ 配下、認証設定など)。
- 現行docsは一部のみ最新化された状態である。参考にしてよいが、
  記述の新旧は見た目で判別できないため、必ず実装と突き合わせて裏を取ること。
- 実装とdocsの記述が食い違う場合、原則として実装を正とする。
  ただし以下のいずれかに該当する場合は「実装側の不具合の可能性」として扱い、
  docsに反映せず、食い違いの内容・該当箇所・判断材料を添えて私に質問すること:
  1. 実装内部に矛盾がある(画面間で同じ項目の挙動が異なる、schemaと実際のクエリが噛み合っていない等)
  2. テストが失敗する、または verify.sh が通らない箇所に関わる
  3. TODO / FIXME / コメントアウトされた処理など、未完成の痕跡がある
  4. docsの記述の方が明らかに一貫していて、実装だけが不自然に逸脱している
- 上記の質問は都度中断せず、リストに蓄積して区切りのよい単位でまとめて
  提示してよい。私の回答が出るまで、該当箇所の新docsは「暫定」と明示して
  先に他の箇所を進めること。
- ファイル構成は現行docsを基本とする。ただし現在の実装と構成が
  合っていないと判断した場合は、変更案を提示して私の承認を得ること。
- 「初版」として書くこと: 旧仕様への言及、変更経緯の記述は一切不要。
  ただし現仕様の理由・根拠(なぜこの設計か)は記載する。
- 実装から仕様が一意に読み取れない箇所は、勝手に補完せず私に質問すること。
- まず1ファイルのみ作成して私の品質確認を受け、承認後に残りを同じ水準で作成する。
- この時点では旧docs・旧plansのファイルは削除しないこと(削除は Step 3 で行う)。
- 新docsは旧docsと明確に区別できるよう、旧ファイルと同名で上書きせず、
  構成変更がない場合も一度私に新旧ファイルの対応表を提示すること。

## Step 2: 新plansの作成
1. まず docs/plans/ 配下の旧plansを全て読み、画面・機能の網羅リストと
   依存関係を整理する(旧plansは新plansの精査元としてのみ使う)。
2. 新plansのファイル一覧(番号+名称)とテンプレ案を提示し、私の承認を得る。
   - 基本は画面単位で1ファイル。画面に紐づかない基盤
     (プロジェクトセットアップ、DBスキーマ、認証、共通レイアウト、シード等)は
     000番台として先頭に置いてよい。
   - ファイル名は「理想的な実装順序」で並ぶよう3桁数字を先頭に付ける。ex: 001-login.md
   - テンプレ: 「目的 / 前提(依存するplan) / 実装内容 / 受け入れ基準 / 検証方法」
3. 承認後、新plansを作成する。全ファイルで体裁を統一すること。
   - 実際の開発で発生したバグ・手戻りの記述は不要。
     最初からバグなく実装できる前提の「初版の計画書」として書く。
   - 旧plansの内容を引き写す際も、旧plansへの言及・参照は書かないこと。
   - この時点では旧plansのファイルは削除しないこと(削除は Step 3 で行う)。

## Step 3: 旧docs・旧plansの削除
- 新docs・新plansの全ファイルについて私の完成承認を得ており、かつ
  「暫定」と明示した箇所が全て解消済みであることを確認してから実施する。
- 旧docsファイル・旧plansファイルをワーキングツリーから完全に削除し、
  最終的なディレクトリ構造(上記)と一致していることを確認して私に報告する。
- 削除対象と残すファイルの一覧を、削除実行前に私に提示して承認を得ること。
- 削除完了後、Step 4 の整合性レビューに進む。

## Step 4: 新docs・新plansの整合性レビュー
- 利用可能なサブエージェント(verifier / code-reviewer 等)が定義されていれば、
  レビューはサブエージェントに実施させること。なければ自身でレビューする。
  いずれの場合もレビュー担当にはファイルの編集をさせず、指摘の報告のみとする。
- レビュー観点:
  1. 新docsと現在の実装(schema.prisma、app/ 配下等)が一致しているか
  2. 新docs同士で矛盾がないか(用語・画面名・テーブル名・権限ルールの不一致等)
  3. 新plansの番号順が依存関係(前提とするplan)と矛盾していないか
  4. 新plansが新docsの仕様と整合しているか、画面・機能に抜け漏れがないか
  5. 旧docs・旧plansへの言及や変更経緯の記述が紛れ込んでいないか
  6. 全plansでテンプレ体裁が統一されているか
  7. 「暫定」表記が残っていないか
- レビュー結果は指摘一覧として私に報告すること。
  客観的な誤り(型の不一致、ファイル間の矛盾、記載漏れ等、正解が一意なもの)は
  修正して報告でよい。仕様の解釈が絡む指摘は判断せず私に質問すること。

## 共通ルール
- 各Stepの完了ごとに報告し、次のStepに進む前に確認を取ること。
- 解釈が分かれる点は判断せず質問すること。
- 旧docs・旧plansが後で必要になった場合は git 履歴から参照できる
  (削除コミット以前の履歴に残る)。

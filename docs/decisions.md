# 主要な設計判断の記録

実装時に「なぜこの仕様か」を確認できるよう、最終的な仕様とその理由を残す。検討過程・改訂の経緯は、本リポジトリの外で別管理している要件定義書（Word）に記録されている（本リポジトリには含めない）。

## 技術スタック

- **Next.js (App Router) + TypeScript + Tailwind CSS**。フロントエンド・バックエンド両方を単一リポジトリで管理し、Server Components / Server Actions を活用する。
- **Prisma ORM (v7)**。スキーマファイル（`prisma/schema.prisma`）をDBの唯一の真実の源とする。`prisma.config.ts`による新config方式を採用。
- **Auth.js v5（next-auth@beta）+ `@auth/prisma-adapter`**。`User`/`Account`/`Session`/`VerificationToken`モデル・テーブル名（複数形）ともAuth.js標準に合わせる（他の業務テーブルの単数形命名とは異なる例外。Auth.js標準準拠を優先）。プロバイダの識別情報は`accounts`テーブル（`provider`/`provider_account_id`）で管理し、`signIn`コールバックで紐付き済みAccountのproviderと今回の認証プロバイダを比較して「プロバイダ不一致」を自前判定する。
  - **セッション戦略は環境で分ける**: 本番はDBセッション、開発環境はJWT。理由: 開発専用のCredentialsプロバイダ（開発用ログイン）がAuth.jsの仕様上JWT戦略でしか動作しないため（`lib/auth.ts`参照）。実SSO実装が完了し開発用ログインを削除した時点で、開発環境もDBセッションに統一できる。
  - **技術的な例外事項**: `users`（Prismaモデル名`User`）のみ、PKを`id`（TEXT/cuid）とする（Auth.jsアダプタの要件。命名規則の`id`＝SERIAL統一の唯一の例外）。`created_by`/`created_program`/`updated_by`/`updated_program`はNULL可（EDT006の管理職操作時のみ設定され、Auth.js自身によるログイン時の`updateUser`呼び出しは監査情報を持たないため）。`accounts`/`sessions`/`verification_tokens`の3テーブルはAuth.jsアダプタが直接read/write/hard-deleteするシステム管理テーブルであり、監査カラム・論理削除のいずれも持たない。
- **Vercel Postgres（Neon、PostgreSQL 16）**。Vercelへのデプロイと同一エコシステム。接続文字列は`DATABASE_URL`（コネクションプーリング用）と`DIRECT_URL`（マイグレーション用直接接続）の2本を使用。

## 命名規則

- **サロゲートキーのPKはカラム名`id`で統一する**。Prismaの慣習に沿い、FKとの読み分け（`skill.id`を`skill_id`が参照）が明快なため。唯一の例外は`users`（Auth.jsアダプタの要件。「技術スタック」参照）。
- **employeeは`id`（SERIAL）をPKとし、業務上の社員ID`employee_id`（VARCHAR(6)、管理職採番）はUNIQUE制約付きの非PKカラムとして持つ**。他テーブルからのFKは`employee.id`ではなく`employee.employee_id`を参照する。理由: `employee_id`はEDT006（新規アカウント登録）・AUTH001（ログイン照合）・経歴書検索など画面上でも直接入力・参照される業務コードであり、FK値がそのまま人間に読める社員IDになる可読性上の利点があるため。

## 認証

- **本番環境ではSSO（Azure AD／Google／GitHub）のみを認証経路とし、ID/パスワード認証は提供しない**。テストは各SSOプロバイダのテスト用アカウントで代替する。
- **開発・テスト環境限定で、社員IDを入力するだけの開発用ログイン（Credentialsプロバイダ）を用意する**。理由: 実SSOプロバイダのアプリ登録・テストテナント整備が整うまでの間も、22画面をロール別に確認する導線を確保するため。`NODE_ENV=production`では当該プロバイダを登録せず無効化し、本番の認証経路には一切影響しない（`lib/auth.ts`参照）。実SSO実装が完了し画面確認導線を代替できる状態になった時点で削除する。
- **新規社員はメールアドレスの事前登録で照合する**。管理職がEDT006で社員ID・メールアドレスだけを先に登録し、本人が初回ログイン時にメールアドレス一致で認証、その場で`accounts`テーブルに今回のプロバイダとの紐付けを作成する。理由: SSOプロバイダのゲスト・個人アカウント等が誤って初回登録フローに入り込むのを防ぐため。
- **Google／GitHubアカウントでのログインにも対応する**。1アカウントにつき初回ログインで使用したプロバイダに固定し、途中でのプロバイダ切替は非対応。**異なるプロバイダで認証された場合はログインさせず「プロバイダ不一致」エラーを表示する**（文言はAUTH001参照）。GitHubは個人アカウントで会社メールと紐付いていない場合、未登録エラーになる制約がある。

## 組織構造

- **事業部＞部署＞Grの3階層を1つの自己参照テーブル（organization_unit）で表現する**。理由: 「事業部のみ所属」「部署までしか所属しない社員」等、所属の深さが社員によって異なるため、3つの固定テーブルでは表現できない。

## 退職者の扱い

- **退職判定は`employee.employment_status`（現職／退職）という専用カラムで行う**。理由: `deleted_at`は本来「レコードの論理削除」という別目的の汎用カラムであり、退職という業務上重要な状態を混在させると意味が読み取りにくくなるため。`deleted_at`はレコード削除の監査用として残し、退職判定には使わない。
- **EDT007に「現職に戻す」操作を設ける**。理由: 誤操作による退職処理の取り消しや再入社に、DB作業なしで画面から対応できるようにするため。退職中のアカウントにのみ表示する。

## スキル・資格・マスタ全般

- **スキル名・資格名・現場ポジション名・現場名はシステム全体でユニーク（カテゴリをまたいでも重複不可）**。理由: 表記ゆれ・重複データ防止を優先。
- **現場名（site）もマスタ化する（MST005 現場マスタ管理）**。理由: 自由記述だと表記ゆれが発生するため。命名は他マスタと同様にシステム全体でユニーク。
- **使用中（他レコードから参照されている）のマスタは削除不可とする**。削除操作時に参照チェックを行い、参照があればエラー表示する。skill_versionの`is_active`（非表示化）は「新規選択を止めたいが過去データは残したい」場合の手段として併存させる。理由: 参照を残したまま削除を許すと経歴書の表示が壊れ、is_active方式への全面統一ではマスタ整理（完全な削除）ができなくなるため。
- **カテゴリはスキルと資格で完全に独立したマスタ**。スキル側でカテゴリを追加しても資格側には影響しない。
- **スキルのバージョンは複数同時登録可能、未入力なら「バージョン管理なし」**。理由: 1スキルに複数バージョン（Java 8/11/17等）を登録するニーズがあるため。
- **project_skill・project_role_linkにも複合ユニーク制約を設ける**（employee_skillと同方針）。同一プロジェクトへの同一スキル・同一役割の重複登録をDB制約で防ぐ。親レコードの論理削除時は従属する子レコードも同一トランザクションで論理削除する（schema.md 一般ルール参照）。
- **`employee_skill.skill_version_id` はNULLで「バージョン管理なし」を表す**。複合ユニーク`(employee_id, skill_id, skill_version_id)`にはPostgreSQL 15+の`UNIQUE NULLS NOT DISTINCT`を適用し、NULL同士も同一視して重複を禁止する。Prisma 7はこの構文をスキーマでネイティブサポートしないため、`prisma migrate dev`で生成したマイグレーションSQLの`CREATE UNIQUE INDEX`末尾に`NULLS NOT DISTINCT`を手動で追記する運用とする。
- **システム全体ユニークな名称（スキル名・資格名・現場ポジション名・現場名）を論理削除済み行と同じ名前で新規登録する場合、削除済み行を復活させる（新規行のINSERTはしない）**。理由: これらのユニーク制約は論理削除(`deleted_at`)を考慮しないDBレベルの制約のため、削除済み行が残っている限り同名の新規作成はユニーク制約違反になる。削除済みでない（現役の）同名行との重複は従来通りエラーで弾く。

## データ移行

- **既存Excelからの移行・インポート機能は提供しない**。新システムでは社員が新規に入力する運用とする。

## PWA方針

- オフライン編集・ホーム画面追加導線・プッシュ通知は**いずれも対応不要**。スマートフォンからの更新のしやすさが目的であり、これらの機能は優先度が低いと判断した。

## 出力（PDF/Excel）

- **PDF出力は「PDF出力ボタン→印刷用プレビュー（REF005）→ダウンロード/印刷」の2段階フローとする**。理由: 出力直前に見た目（レイアウト）を確認できるようにするため。マイページ・経歴書詳細画面に出力ボタンを統合し、独立画面は持たない。Excelは画面遷移なしの直接ダウンロードとする。

## 経験年数の計算

- **経験年数は全プロジェクト期間の和集合（重複期間は1回）を月数換算し、12で割った整数部とする**。理由: プロジェクトの掛け持ち（期間重複）がある場合、単純合算では実経験年数を超えてしまうため。進行中（`end_date=NULL`）のプロジェクトは計算時点の年月まで含める。再計算のタイミングはプロジェクト経歴の登録・更新・削除時。

## ロールと画面の整合

- **一般社員も閲覧範囲内の社員の経歴書を閲覧できる**。理由: 部署内での相互参照（経歴書の書き方の参考、チーム内のスキル把握）を可能にするため。判定は原則「遡った部署（unit_level=2）の一致」だが、**事業部直下所属の社員が絡む場合は事業部（unit_level=1）の一致で判定**する（事業部付スタッフ等が誰とも相互閲覧できなくなるのを防ぐため）。未所属（NULL）は対象外。判定ルールの詳細はscreens.mdのREF002。編集は従来どおり本人（＋管理職の自分の分）のみ。
- **REF008（スキルマップ／組織ダッシュボード）は全ロール閲覧可能で、保有者名も全ロールに表示する**。組織単位ごとのスキル・資格保有状況を可視化するために追加した機能。理由: 「誰がどのスキルを持つか」は社内で共有してよい情報と判断したため。ただし保有者名から経歴書詳細（REF003）へ遷移できるかは閲覧範囲の判定ルールに従う。
- **人事・営業は自分の経歴書を作成しない（マイページなし）**。理由: 人事・営業の職務は他社員の経歴書の検索・閲覧であり、自身の経歴書登録のニーズはないため。マイページタイルの表示対象は一般社員／管理職のみ。これに伴い、人事・営業の初回ログインはEDT001を経ずREF001へ直行し、`is_registered`はログイン成立時に自動でTRUEとする。
- **EDT003（スキル登録）／EDT004（資格登録）のカテゴリは既存マスタからの選択のみ**。理由: マスタ管理は管理職専用（MST001/002）という原則と一致させるため。カテゴリの追加が必要な場合は管理職に依頼する運用とする。

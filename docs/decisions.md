# 主要な設計判断の記録

実装時に「なぜこの仕様か」を確認できるよう、経緯を残す。より詳細な検討過程は、本リポジトリの外で別管理している要件定義書（Word）に記録されている（本リポジトリには含めていない）。

## 技術スタック

- **Next.js (App Router) + TypeScript + Tailwind CSS**。フロントエンド・バックエンド両方を単一リポジトリで管理。App Router を採用し、Server Components / Server Actions を活用する方針。
- **Prisma ORM (v7)**。スキーマファイル（`prisma/schema.prisma`）をDBの唯一の真実の源とする。Prisma 6+ の新 config 方式（`prisma.config.ts`）を採用。
- **Auth.js v5（next-auth@beta）、セッション戦略は DB セッション（`@auth/prisma-adapter` 採用、2026-07再改訂）**。当初は`user_account`の独自1テーブル設計とアダプタ標準スキーマ（User/Account/Session/VerificationToken）の不一致、および「プロバイダ不一致はエラーとする」仕様とアダプタの自動アカウント紐付け機能が噛み合わないことを理由にJWT戦略を採用していた。しかし`user_account`をAuth.js標準の`User`モデルへ統合し、テーブル名も`users`にリネームしてモデル名・テーブル名ともAuth.js標準に完全に合わせた（`accounts`/`sessions`/`verification_tokens`も同様、複数形はAuth.js公式ドキュメントの推奨命名。他の業務テーブルの単数形命名とは異なるが、Auth.js標準準拠を優先した例外）。`external_id`/`auth_provider`カラムを`accounts`テーブル（`provider`/`provider_account_id`）に置き換えることで、当初の不一致は解消できると判断した。「プロバイダ不一致はエラー」は、`signIn`コールバックで紐付き済みAccountのproviderと今回の認証プロバイダを比較して自前判定する（アダプタの自動紐付け自体は妨げない）。**Credentialsプロバイダ（開発用ログイン）はAuth.jsの仕様上JWT戦略でしか動作しないため、開発環境のみJWT・本番環境はDBセッションと環境でセッション戦略を分けている**（`lib/auth.ts`参照）。実SSO実装が完了し開発用ログインを削除した時点で、開発環境もDBセッションに統一できる。
  - **技術的な例外事項**: `users`（Prismaモデル名`User`）のみ、PKを`id`（TEXT/cuid）とする（Auth.jsアダプタの要件。命名規則の`id`＝SERIAL統一の唯一の例外）。また`created_by`/`created_program`/`updated_by`/`updated_program`はNULL可とする（EDT006の管理職操作時のみ設定され、Auth.js自身によるログイン時の`updateUser`呼び出しは監査情報を持たないため）。`accounts`/`sessions`/`verification_tokens`の3テーブルはAuth.jsアダプタが直接read/write/hard-deleteするシステム管理テーブルであり、監査カラム（created_at等）・論理削除（deleted_at）のいずれも持たない。
- **Vercel Postgres（Neon、PostgreSQL 16）**。Vercel へのデプロイと同一エコシステム。接続文字列は `DATABASE_URL`（コネクションプーリング用）と `DIRECT_URL`（マイグレーション用直接接続）の2本を使用。

## 命名規則（2026-07 確定）

- **サロゲートキーのPKはカラム名`id`で統一する**。従来は`id`と`<テーブル名>_id`が混在していたが、Prismaの慣習に沿い、FKとの読み分け（`skill.id`を`skill_id`が参照）も明快なため`id`に統一した。実装前のため変更コストなし。**唯一の例外は`users`**（Auth.jsアダプタの要件でTEXT/cuid。詳細は「技術スタック」参照）。
- **employeeは`id`（SERIAL）をPKとし、業務上の社員ID`employee_id`（VARCHAR(6)、管理職採番）はUNIQUE制約付きの非PKカラムとして残す（2026-07再改訂）**。当初は`employee_id`をそのままPKとする例外としていたが、「PKはid統一」という規則の例外を極力減らす方針に変更し、サロゲートキーを導入した。ただし他テーブルからのFKは`employee.id`ではなく引き続き`employee.employee_id`を参照する。理由: `employee_id`は管理職が新規登録時に一度だけ採番する安定した識別子であり、EDT006（新規アカウント登録）・AUTH001（ログイン照合）・経歴書検索など画面上でも直接入力・参照される業務コードである。他テーブルのFK値がそのまま人間に読める社員IDになる可読性上の利点を維持するため、FK参照先は変更しなかった。

## 認証

- **本番環境ではSSO（Azure AD／Google／GitHub）のみを認証経路とし、ID/パスワード認証は提供しない**。テストは各SSOプロバイダのテスト用アカウントで代替する。
- **開発・テスト環境限定で、社員IDを入力するだけの開発用ログイン（Credentialsプロバイダ）を用意する（2026-07改訂、運用方針を再度変更）**。当初はテスト・開発環境限定のID/パスワードログインを用意する方針だったが、運用負荷を理由に一度廃止した。しかし実SSOプロバイダのアプリ登録・テストテナント整備が整うまでの間、22画面をロール別に確認する導線が失われるため、開発用ログインを復活させた。`NODE_ENV=production`では当該プロバイダを登録せず無効化し、本番の認証経路には一切影響しない（`lib/auth.ts`参照）。将来、実SSO実装が完了し画面確認導線を代替できる状態になった時点で削除する。
- **新規社員はメールアドレスの事前登録で照合する**。管理職がEDT006で社員ID・メールアドレスだけを先に登録し、本人が初回ログイン時にメールアドレス一致で認証、その場で`accounts`テーブルに今回のプロバイダとの紐付け（`provider`/`provider_account_id`）を作成する。理由: SSOプロバイダのゲスト・個人アカウント等が誤って初回登録フローに入り込むのを防ぐため。
- **Google／GitHubアカウントでのログインにも対応する**。プロバイダの識別情報は`accounts`テーブル（Auth.js標準）の`provider`/`provider_account_id`で管理し、特定プロバイダに依存しない。1アカウントにつき初回ログインで使用したプロバイダに固定し、途中でのプロバイダ切替は非対応。**異なるプロバイダで認証された場合はログインさせず「プロバイダ不一致」エラーを表示する**（挙動が未定義だったため実装可能なエラー仕様として確定。文言はAUTH001参照）。GitHubは個人アカウントで会社メールと紐付いていない場合、未登録エラーになる制約がある。

## 組織構造

- **事業部＞部署＞Grの3階層を1つの自己参照テーブル（organization_unit）で表現する**。当初「department」という単一テーブルで検討したが、「事業部のみ所属」「部署までしか所属しない社員」等、所属の深さが人によって異なることが判明し、3つの固定テーブルでは矛盾が生じるため自己参照方式に変更した。

## 退職者の扱い

- **退職判定は`employee.employment_status`（現職／退職）という専用カラムで行う**。当初は`users.deleted_at`（論理削除の日付）を退職判定に流用する方針だったが、最終的に専用カラムに変更した。理由: `deleted_at`は本来「レコードの論理削除」という別目的の汎用カラムであり、退職という業務上重要な状態を汎用カラムに混在させると意味が読み取りにくくなるため。`deleted_at`はレコード削除の監査用として残し、退職判定には使わない。
- **EDT007に「現職に戻す」操作を設ける（2026-07確定）**。誤操作による退職処理の取り消しや再入社に、DB作業なしで画面から対応できるようにするため。退職中のアカウントにのみ表示する。

## スキル・資格・マスタ全般

- **スキル名・資格名・現場ポジション名・現場名はシステム全体でユニーク（カテゴリをまたいでも重複不可）**。表記ゆれ・重複データ防止を優先した。一時的にユニーク制約なしの方針を検討したが、各テーブル内で重複を許容したくないという要件により、最終的に元の「システム全体でユニーク」に確定した。
- **使用中（他レコードから参照されている）のマスタは削除不可とする（2026-07確定）**。削除操作時に参照チェックを行い、参照があればエラー表示する。skill_versionの`is_active`（非表示化）は「新規選択を止めたいが過去データは残したい」場合の手段として併存させる。理由: 参照を残したまま削除を許すと経歴書の表示が壊れ、is_active方式への全面統一はマスタ整理（完全な削除）ができなくなるため。
- **カテゴリはスキルと資格で完全に独立したマスタ**。スキル側でカテゴリを追加しても資格側には影響しない。
- **スキルのバージョンは複数同時登録可能、未入力なら「バージョン管理なし」**。当初は「バージョン管理チェックボックス＋単一バージョン入力」だったが、実際には1スキルに複数バージョン（Java 8/11/17等）を登録するニーズがあるため変更した。
- **project_skill・project_role_linkにも複合ユニーク制約を設ける**（employee_skillと同方針）。同一プロジェクトへの同一スキル・同一役割の重複登録をDB制約で防ぐ。親レコードの論理削除時は従属する子レコードも同一トランザクションで論理削除する（schema.md 一般ルール参照）。
- **`employee_skill.skill_version_id` は sentinel 0 ではなく NULL で「バージョン管理なし」を表す**。元仕様では `skill_version_id=0`（無効な FK 値）を番兵値として使う設計だったが、Prisma の FK 制約と相性が悪いため `NULL` 可に変更した。複合ユニーク `(employee_id, skill_id, skill_version_id)` には PostgreSQL 15+ の `UNIQUE NULLS NOT DISTINCT` を適用し、`NULL` 同士も同一視して重複を禁止する。Prisma 7 はこの構文をスキーマでネイティブサポートしないため、`prisma migrate dev` で生成したマイグレーション SQL の `CREATE UNIQUE INDEX` 末尾に `NULLS NOT DISTINCT` を手動で追記する運用とする。

## データ移行

- **既存Excelからの移行・インポート機能は提供しない**。当初は「一括インポート機能を作る」方針だったが、最終的に不要と判断し、新システムでは社員が新規に入力する運用に変更した。

## PWA方針

- オフライン編集・ホーム画面追加導線・プッシュ通知は**いずれも対応不要**。スマートフォンからの更新のしやすさが目的であり、これらの機能は優先度が低いと判断した。

## 画面構成の変遷

- 当初PDF/Excel出力は独立画面（旧OUT001）だったが、マイページ・経歴書詳細に出力ボタンを統合し、「PDF出力ボタン→印刷用プレビュー（REF005）→ダウンロード/印刷」の2段階フローに変更した。
- 現場名（site）は当初自由記述だったが、表記ゆれ防止のためマスタ化した（MST005 現場マスタ管理を新設）。命名は他マスタと同様にシステム全体でユニーク。
- スキルマップ／組織ダッシュボード（REF008）は、組織単位ごとのスキル・資格保有状況を可視化するために追加した機能。

## 経験年数の計算（2026-07 確定）

- **経験年数は全プロジェクト期間の和集合（重複期間は1回）を月数換算し、12で割った整数部とする**。プロジェクトの掛け持ち（期間重複）がある場合、単純合算では実経験年数を超えてしまうため。進行中（`end_date=NULL`）のプロジェクトは計算時点の年月まで含める。再計算のタイミングはプロジェクト経歴の登録・更新・削除時。

## ロールと画面の整合（2026-07 確定）

- **一般社員も閲覧範囲内の社員の経歴書を閲覧できる（2026-07 仕様変更）**。当初は「自分の経歴書のみ」だったが、部署内での相互参照（経歴書の書き方の参考、チーム内のスキル把握）を可能にするため開放した。判定は原則「遡った部署（unit_level=2）の一致」だが、**事業部直下所属の社員が絡む場合は事業部（unit_level=1）の一致で判定**する（事業部付スタッフ等が誰とも相互閲覧できなくなるのを防ぐため。2026-07確認済み）。未所属（NULL）は対象外。判定ルールの詳細はscreens.mdのREF002。編集は従来どおり本人（＋管理職の自分の分）のみ。
- **REF008（スキルマップ）は全ロール閲覧可能で、保有者名も全ロールに表示する（2026-07 仕様変更）**。当初は管理職のみ→一時は保有者名を閲覧権限でマスキングする案も検討したが、「誰がどのスキルを持つか」は社内で共有してよい情報と判断し、氏名まで全表示に確定した（2026-07確認済み）。ただし保有者名から経歴書詳細（REF003）へ遷移できるかは閲覧範囲の判定ルールに従う。

- **人事・営業は自分の経歴書を作成しない（マイページなし）**。当初REF001では「マイページタイルは全ロール共通」としていたが、人事・営業の職務は他社員の経歴書の検索・閲覧であり、自身の経歴書登録のニーズはないと判断し、マイページタイルの表示対象を一般社員／管理職に限定した。これに伴い、人事・営業の初回ログインはEDT001を経ずREF001へ直行し、`is_registered`はログイン成立時に自動でTRUEとする。
- **EDT003（スキル登録）／EDT004（資格登録）のカテゴリは既存マスタからの選択のみ**。当初は新規入力候補の表示も許す記述だったが、マスタ管理は管理職専用（MST001/002）という原則と矛盾するため、選択のみに確定した。カテゴリの追加が必要な場合は管理職に依頼する運用とする。

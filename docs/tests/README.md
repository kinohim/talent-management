現行のテスト実装（Vitest 単体テスト）に対応する仕様書。機能カテゴリ別に分割しており、「どの関数のどの観点がテスト済みか」を一覧できる。テストコードを変更したら、対応するカテゴリのファイルも更新すること。

## テスト方針

### 対象範囲

- テストは **`lib/` 層の単体テストのみ**。テストファイルは対象と同じディレクトリに `*.test.ts` として置く（CLAUDE.md のテスト方針どおり）
- Server Actions・Route Handler・React コンポーネントを直接テストするのではなく、**ロジックを `lib/` の純関数に切り出して単体テストする**
- 実行環境は node（jsdom 不使用）。DOM を伴うコンポーネントテストは対象外
- カバレッジ計測は設定していない

### モック方針

- DB 依存の関数は `vi.mock("@/lib/prisma")` で Prisma Client をモックし、実際の Neon には接続しない。`beforeEach` で `mockReset()` し、`mockResolvedValue` で戻り値を注入する
- 認可ガード等は `vi.mock` で `next/navigation`（`redirect` をスロー化）や `@/lib/auth` を差し替える
- 純粋関数はモックなしで入力→出力を直接検証する

### 命名規約

- `describe` = テスト対象の関数名（集計ロジック等は日本語の機能名も可）
- `it` = 日本語で期待動作を記述
- 不具合修正に伴う再発防止テストには `回帰:` プレフィックスを付ける

### 仕様書の記載粒度

- 節（`##`）はトップレベル `describe` 単位。「`対象: <ソース> / テスト: <テストファイル>`」「`概要:`」「`前提:`」の3行を記した上で、`it` 1件 = 1行の観点表（No／確認観点／分類）を置く
- 分類は「正常系／異常系／境界値／回帰」の4種
- 入力データ・期待値の詳細は書かない（テストコード自体を正とする）

## 実行方法

```
npm run test             # 全テスト実行 (vitest run)
npx vitest run <path>    # 特定ファイルのみ実行
npm run verify           # フル検証 (prisma generate + lint + 型 + テスト)
```

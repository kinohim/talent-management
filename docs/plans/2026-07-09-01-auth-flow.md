# 実行計画①: AUTH001(ログイン) / REF001(トップ) / EDT001(基本情報登録)

## Context

現時点でDBスキーマ・Prismaシングルトン・Auth.js基盤(開発用ログインのみ)は整備済みだが、
画面は`app/page.tsx`の「準備中」プレースホルダーのみで、docs/screens.mdの22画面はまだ
1つも本実装されていない。画面遷移図(screen-flow.md)では**AUTH001(ログイン)→REF001(トップ)
→EDT001(初回登録)**が全画面の起点であり、ここが未実装だと他のどの画面にも到達できない。
また現状はNextAuth標準の英語ログイン画面(`/api/auth/signin`)が露出しており、
docs/README.mdの「ライブラリのデフォルト画面は自前の日本語画面に差し替える」という
明示ルールに違反した状態にある。

この3画面をひとつの検証可能な認証フローとして実装し、以後の画面(REF002〜REF008、
EDT002〜EDT007等)を追加していく土台を作る。

**今回のスコープ外の扱い(人間に確認済み)**:
- EDT001保存後の遷移先は仕様上REF004(マイページ)だが、REF004は未実装のため
  **一時的にREF001(トップ)へ戻す**。コード上にTODOコメントで明記し、REF004実装時に差し替える。
- REF001のうち「経歴書一覧」「アカウント一覧」「マスタ管理」「スキルマップ」タイルは
  ロール別の表示条件だけ実装し、**クリック不可・「準備中」ラベル**でグレーアウト表示する。
- 実SSO(Azure AD/Google/GitHub)は未実装のまま。AUTH001の画面には開発用ログイン
  (社員IDログイン)のUIを表示する。将来SSO実装時に差し替える前提のコメントを残す。
- プロバイダ不一致エラーは現状1プロバイダのみで到達不可のため、エラーコード・文言だけ
  用意し、実際にthrowする分岐は実装しない。

## 実装方針

### ルーティング構成

```
app/
  login/
    page.tsx                         # AUTH001
    actions.ts                       # loginAction (Server Action)
    error.tsx
  (authenticated)/
    layout.tsx                       # 未ログイン→/loginへredirect。共通ヘッダー(氏名/ロール/ログアウト)
    page.tsx                         # REF001 (既存app/page.tsxを置き換え、旧ファイルは削除)
    loading.tsx
    error.tsx
    register/
      page.tsx                       # EDT001
      actions.ts                     # saveBasicInfo (Server Action)
```

ルートグループ`(authenticated)`はURLに影響しない(`/`, `/register`のまま)。REF001・EDT001に
共通の「未ログインなら`/login`」ガードと共通ヘッダーを`layout.tsx`に一箇所化する。
`.claude/skills/new-screen/SKILL.md`が要求する「画面ごとの認可チェック」は各`page.tsx`側の
ロール別リダイレクトで満たす(layoutは土台、pageは画面固有チェックという役割分担)。

認可方式は`middleware.ts`(Edge runtime)ではなく各`page.tsx`/`layout.tsx`での`auth()`チェックを
採用する。理由: `lib/auth.ts`は`PrismaAdapter`+`@prisma/adapter-neon`を使うため、
middleware(既定でEdge runtime)に載せるにはランタイム整合性の追加検証が必要でコストが高い。
`new-screen`スキルが要求する方式そのものでもある。対象ページが増えてmiddleware化の
投資対効果が上がった時点で再検討する。

`isRegistered`はセッション(JWT/DBセッション)には持たせない。EDT001保存直後にJWTが
再生成されない不整合を避けるため、`lib/auth-routing.ts`でPrismaから都度取得する。

### `lib/auth.ts` / `lib/dev-login.ts` の変更

- `lib/dev-login.ts`: `findDevLoginUser`の戻り値を判別可能なユニオン型
  `{ok:true,user}|{ok:false,reason:"EMPTY_EMPLOYEE_ID"|"NOT_REGISTERED"|"RETIRED"}`に変更し、
  「未登録」と「退職済み」を区別できるようにする(現状はどちらもnullで区別不可)。
- `lib/auth-errors.ts`(新規): `next-auth`の`CredentialsSignin`をサブクラス化した
  `NotRegisteredError`/`RetiredEmployeeError`/`ProviderMismatchError`(コードのみ、今回はthrowしない)。
  `messageForLoginError(error)`でAUTH001所定の日本語文言に変換する。
  - 未登録: 「このメールアドレスは登録されていません。管理者に新規登録を依頼してください。」
  - 退職済み: 「このアカウントは無効化されています。心当たりがない場合は管理者にお問い合わせください。」
  - プロバイダ不一致: 「このアカウントは別のログイン方法で登録されています。初回に使用したログイン方法をお使いください。」(未到達コード)
- `lib/env.ts`(新規): `isProduction`を`lib/auth.ts`から切り出し、ログインページ側でも再利用。
- `lib/auth.ts`: `authorize()`内で上記エラーをthrow。`events.signIn`を追加し、
  ロールが`HR_SALES`かつ`isRegistered=false`のログイン成立時に`employee.isRegistered`を
  自動でTRUEに更新する(docs/screens.md AUTH001「人事・営業は経歴書を作成しないため
  EDT001を経ずREF001へ直行」に対応)。`employee.name`へのSSO表示名設定は、
  開発用ログインには対応する情報源がないため今回は実装しない(実SSO実装時に対応)。

### 認可・画面遷移ロジック

`lib/auth-routing.ts`(新規): `isEmployeeRegistered(employeeId)`と
`resolveDestination(user)`(HR_SALESなら常に`/`、それ以外は`isRegistered`で`/`か`/register`)を提供。
ログイン直後の遷移判定と、REF001自体を開いたときの「未登録なら/registerへ戻す」恒常ガードの
両方で使う。

### EDT001 Server Action設計

- `zod`をpackage.jsonに明示的な依存として追加する(`npm install zod`。現状は推移的依存のみで
  直接依存ではないため)。
- `lib/basic-info-schema.ts`(新規): `basicInfoSchema`(zod)、`parseBasicInfoForm(formData)`、
  `flattenFieldErrors(error)`。氏名/カナ50文字、カナは全角カタカナ正規表現、路線名・駅名・
  学校名・学部学科名100文字、日付/年月は文字列パターンのみ検証(変換はしない)。
- `lib/date-format.ts`(新規): `toDateInputValue`/`toMonthInputValue`(表示用)、
  `parseDateOnly`/`parseYearMonth`(保存用、UTC基準で日付ズレを避ける)。
  `graduationYearMonth`はdocs/schema.md規約に従いYYYYMM01形式で保存。
- `lib/organization-unit.ts`(新規): `getOrganizationUnitOptions()`(全件取得)、
  `resolveSelectionFromLeaf(units, leafId)`(保存済み最下層idから3セレクトの初期値を逆算)、
  `resolveOrganizationUnitId(selection)`(送信された最下層id優先で採用。祖先関係の再検証はしない
  ——特権境界ではなく本人の自己申告メタデータのため、idの実在・未削除チェックのみ行う)。
- `lib/role-label.ts`(新規): `roleLabel(role)`で3ロールの日本語ラベルを返す(ヘッダー表示用)。
- `app/(authenticated)/register/actions.ts`: `saveBasicInfo(prevState, formData)`。
  `auth()`で認証・HR_SALES拒否 → zodパース → `resolveOrganizationUnitId` →
  `prisma.employee.update`(`isRegistered:true`, `updatedBy/updatedProgram`付き) →
  `redirect("/")`(REF004実装までの暫定遷移、コメントで明記)。
  社員ID・メールアドレスはフォームから受け取らず、常に`session.user.employeeId`を信頼する。
- `app/(authenticated)/register/page.tsx`: `auth()`チェック→HR_SALESなら`/`へredirect→
  `employee`と`organizationUnit`一覧を取得→`BasicInfoForm`に渡す。

### 事業部/部署/Gr 3階層セレクト

Server Component(`register/page.tsx`)で`OrganizationUnit`全件を取得し、Client Component
(`components/basic-info/OrganizationUnitSelect.tsx`)にpropsで渡してメモリ内フィルタする
(選択都度のサーバー問い合わせは行わない)。運用規模(MST004は手動登録運用、初期データなし)から
件数は小〜中規模と想定され、全件クライアント側フィルタで十分。事業部選択→部署選択肢を
`parentId`で絞込み、部署選択→Gr選択肢を絞込み。`<select name="divisionId|departmentId|groupId">`
のネイティブフォーム送出でServer ActionのFormDataに自動的に乗る。

## 実装結果(実施済み)

上記プランに沿って実装し、以下の点は実施の過程で調整した。

- `lib/auth.ts`にAuth.js標準signin画面をAUTH001へ差し替えるための`pages: { signIn: "/login" }`を追加(プラン記載時は未検討だった小さな補完)
- `CredentialsSignin`のimportは`next-auth`からではなく`@auth/core/errors`から行うよう変更
  (`next-auth`経由だとVitest実行時に`next/server`解決エラーになるため。`@auth/core`を
  package.jsonへ明示的な直接依存として追加した)
- `"use server"`ファイルから`initialLoginFormState`/`initialBasicInfoFormState`という
  オブジェクト定数をexportしていたため実行時エラー(「関数以外はexportできない」)が発生。
  各初期値をコンポーネント側(`LoginForm.tsx`/`BasicInfoForm.tsx`)に移して解消(手動確認で発見)
- `npm run verify`(43件→当時39件のテスト)通過、Playwrightでの実ブラウザ操作により
  未登録/退職済みエラー、ロール別タイル表示、EDT001保存、人事・営業の自動登録、
  ロールガードの10シナリオを確認済み

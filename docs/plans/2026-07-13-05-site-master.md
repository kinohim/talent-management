# 実行計画: MST005(現場マスタ管理)

## Context

MST004(部署マスタ管理)・MST001(スキル)・MST002(資格)・MST003(現場ポジション)に続き、マスタ管理系の最後(5画面目)としてMST005(現場マスタ管理)を実装した。`site`マスタ(現場名のみ、カテゴリなし)を管理職が登録・編集・削除する画面で、構造はMST003(現場ポジション)とほぼ同一。これでマスタ管理系5画面(MST001〜005)がすべて揃った。

## 実装ファイル

### 新規
- `app/(authenticated)/master/sites/page.tsx`: MANAGER専用。現場一覧を取得し`SiteMasterManager`へ渡す
- `app/(authenticated)/master/sites/actions.ts`: `saveSite(siteId: number | null, prevState, formData)`・`deleteSite(siteId)`
- `components/master/SiteMasterManager.tsx`: 追加フォーム(現場名のみ)+一覧のルート("use client"。`ProjectRoleMasterManager.tsx`と同型)
- `components/master/SiteMasterRow.tsx`: 行の表示/編集/削除("use client"。`ProjectRoleMasterRow.tsx`と同型)
- `lib/site-master-schema.ts` + テスト: 現場名のzodバリデーション(1〜100文字)
- `lib/site-master.ts` + テスト: `getSiteDeleteBlockReason(siteId)`(`Project.siteId`の`deletedAt: null`件数)

### 変更
- `app/(authenticated)/master/page.tsx`: `MASTER_TILES`の`sites`エントリに`href: "/master/sites"`を追加
- `lib/breadcrumbs.ts`: `"/master/sites"`エントリを追加

### 再利用(変更なし)
- `lib/auth-guards.ts`の`requireManager()`
- `lib/prisma-errors.ts`の`isUniqueConstraintViolation()`
- `components/ui/ConfirmDialog.tsx`(CMN001)

## 設計判断

1. **削除時参照チェックは最初から`deletedAt: null`で絞り込む**: MST001〜003で見つかった「論理削除済みプロジェクトの残骸を使用中と誤判定する」バグ(`docs/plans/2026-07-13-04-project-role-master.md`参照)を踏まえ、`getSiteDeleteBlockReason`は最初から`prisma.project.count({ where: { siteId, deletedAt: null } })`で実装した。
2. **論理削除済み同名行の復活ロジックも最初から組み込む**: MST001/002/003で追って修正した「論理削除済み同名行と同じ名前で新規登録すると失敗する」バグ(`site_name`の`@unique`制約は`deleted_at`を考慮しないため)についても、`saveSite`の新規作成分岐に最初から反映した(同名の論理削除済み`Site`があれば`update`で復活、無ければ`create`)。
3. **保存方式**: 他のマスタ画面と同様`revalidatePath`で同一画面に留まる。

## 検証方法

- `npm run verify`
- Playwright(開発用ログイン、管理職`000001`)で以下を確認:
  1. トップ→マスタ管理→現場マスタ管理と辿れ、パンくず・戻る導線が機能する
  2. 現場を新規追加できる(フォームもリセットされる)
  3. 現場名の重複登録で「既に登録されている現場名です。」が表示される
  4. 参照されていない現場の削除に成功する
  5. 削除→同じ名前で再度新規追加が成功する(復活ロジックの確認)
  6. 参照されている現場(EDT005で一時的にプロジェクト経歴の現場として選択して検証)の削除がブロックされ「使用中のため削除できません」が表示される
  7. 一般社員での直接アクセスがトップへリダイレクトされる

## 実装結果(実施済み)

上記プラン通りに実装した。仕様(docs/screens.md)との差異はなし。

### 検証結果

`npm run verify`(lint・tsc・vitest 219件)が通過。Playwrightで以下をすべて確認した。

1. トップ→マスタ管理→現場マスタ管理のパンくず・戻る導線が正しく機能した
2. 「C社在庫管理システム刷新」を新規追加でき、フォームもリセットされた
3. 「A社基幹システム更改」の重複登録を試み「既に登録されている現場名です。」が表示された
4. 未参照の「C社在庫管理システム刷新」の削除に成功した
5. 削除後、同じ名前で再度新規追加するとエラーにならず成功した(復活ロジックが正しく動作)
6. 参照済みの「C社在庫管理システム刷新」(EDT005で一時的にプロジェクト経歴の現場として選択して検証)の削除を試みると「使用中のため削除できません」が表示され、削除がブロックされた
7. 一般社員(`000002`)での`/master/sites`直接アクセスがトップへリダイレクトされた

検証用に一時登録したプロジェクト経歴・現場データはすべて削除して後片付けした。これでマスタ管理系(MST001〜005)がすべて完了した。

# REF008(スキルマップ/組織ダッシュボード)実装計画

## Context

REF001(トップ)の「スキルマップ/組織ダッシュボード」タイルは現在`href`未設定(「準備中」表示)。REF002(経歴書一覧)実装に続き、合意した実装順序に従いREF008を実装する。

REF008はREF002と異なり検索の軸が逆で、「組織単位を起点にスキル・資格側から保有者数を集計する」画面。`docs/decisions.md`に明示的な設計判断があり、**全ロールが閲覧可能で保有者名も全ロールに表示するが、保有者名からREF003(経歴書詳細)へ遷移できるかは既存の閲覧範囲判定ルールに従う**という二層構造になっている。

## 画面仕様(docs/screens.md REF008、161〜170行目)

| 項目 | 種別 | 内容 |
|---|---|---|
| 組織単位選択 | 選択 | 事業部／部署／Grのいずれかを選択(配下含めて集計)。REF002の複数チェックボックスと異なり単一選択 |
| スキル別保有者数集計 | 表示 | スキル名ごとの保有者数(カテゴリ別グルーピング可) |
| 資格別保有者数集計 | 表示 | 資格名ごとの保有者数 |
| 保有者名一覧 | 表示 | 集計項目をクリックすると該当社員名を展開表示(全ロールに氏名を表示)。退職者は集計対象外 |

ユーザーとの確認で以下の未規定点を決定した:
- **組織単位未選択時の初期表示**: 全社集計をデフォルト表示する(選択すると配下限定に絞り込まれる)
- **閲覧範囲外の保有者名**: 氏名はそのまま(リンクなしのテキストとして)表示する。装飾上の差別化は行わない

その他、仕様書に明記のない点は以下の通り解釈する(低リスクな補足):
- 資格側のカテゴリ別グルーピングは明記がないため行わない(スキルのみグルーピング)
- スキルの習熟度(◎/○/△)別内訳は「保有者数」としか書かれていないため出さない(単純な人数集計)
- 「保有者数」は人数であり、資格の重複取得(再取得で行が増える)があっても社員単位で重複排除して数える
- 保有者が0人のスキル/資格は集計行として表示しない(全マスタを列挙するとノイズになるため)

## 閲覧範囲ロジックの再利用方針

REF002で実装済みの以下の関数をそのまま再利用する(新規ロジック追加なし):

- `lib/organization-unit.ts`の`getOrganizationUnitOptions`(組織単位フラット一覧取得)
- `lib/organization-unit-tree.ts`の`buildOrganizationUnitTree`(選択UI用ツリー構築)・`collectDescendantIds`(選択した組織単位+配下idのSet化。「配下含めて集計」の要件そのもの)
- `lib/organization-unit.ts`の`canViewEmployeeResume`(本人/HR_SALES/MANAGERは常にtrue、一般社員は`isWithinResumeViewScope`の閲覧範囲ルールに従う) — 保有者名→REF003リンクの活性/非活性判定にそのまま使う

REF002と異なり、**検索対象の母集団自体はロールで絞り込まない**(全ロールが全社員を集計対象にできる)。ロール制限がかかるのは「保有者名をREF003へのリンクにするかどうか」の1点のみ。

## 実装ファイル

### 新規
- `app/(authenticated)/skill-map/page.tsx`: REF008本体。認可(`resolveDestination`で登録済みガード、REF002/REF003と同じ既存パターン。HR_SALESは常に`"/"`を返すため弾き出しガード不要)→組織単位選択のパース→対象社員+スキル/資格の取得→集計→表示
- `lib/skill-map.ts` + テスト: Prisma非依存の純粋集計関数群
  - `parseSkillMapUnitId(searchParams): number | null`(`unitId`パラメータのパース。`lib/resume-search.ts`の`toIdArray`等と同系統)
  - `aggregateSkillHolders(employees, skillMasterInfo): SkillCategoryGroup[]`: 各社員が保有するスキルid配列(同一スキルの複数バージョンは事前に重複排除しておく)から、スキルごとに保有者リストを集約し、カテゴリ名→スキル名の順でソートしてグルーピングした配列を返す。保有者0人のスキルは含めない
  - `aggregateCertificationHolders(employees, certificationMasterInfo): CertificationHolderRow[]`: 資格ごとに保有者リストを集約し資格名昇順でソート(カテゴリグルーピングなし)。保有者0人の資格は含めない
- `components/skill-map/OrganizationUnitRadioFilter.tsx`("use client"): 組織単位の単一選択(ラジオボタン、階層インデント表示)。先頭に「全社」を選ぶと`unitId`なしで送信。`components/accounts/OrganizationUnitTreeFilter.tsx`と同じ階層インデントパターン(`DEPTH_PADDING`)を踏襲しつつチェックボックス→ラジオに変更、送信は`ResumeFilterForm.tsx`と同じく`router.push`でURL更新
- `components/skill-map/SkillHolderList.tsx`: スキル/資格共通の集計結果表示コンポーネント(Server Component)。カテゴリ見出し(スキルのみ)→各項目を「項目名(N名)」+`<details>/<summary>`で保有者名を展開表示するバー付き行として描画。バーはTailwindのみ(`<div style={{width: \`${pct}%\`}}>`)で保有率を表現(母数=集計対象社員数)。保有者名は`canView`フラグに応じて`<Link href="/resumes/${employeeId}">`かプレーンテキストを出し分け

### 変更
- `components/home/HomeTiles.tsx`: `skill-map`タイルに`href: "/skill-map"`を追加(「準備中」表示を解除)
- `lib/breadcrumbs.ts`: `"/skill-map": { label: "スキルマップ/組織ダッシュボード", parentPath: "/" }`を追加

## ページ側の実装フロー(`app/(authenticated)/skill-map/page.tsx`)

1. `auth()` → 未ログインなら`/login`へ、`resolveDestination`が`"/"`以外ならそちらへリダイレクト(REF002/REF003と同じ既存ガード)
2. `getOrganizationUnitOptions()`でunits取得、`parseSkillMapUnitId`でURLの`unitId`をパース
3. `unitId`が指定されていれば`collectDescendantIds(units, [unitId])`で対象組織単位idのSetを作り、`employee.organizationUnitId in [...ids]`条件を追加。未指定なら組織単位での絞り込みなし(全社)
4. 対象社員を1クエリで取得: `employmentStatus: ACTIVE`(退職者除外)・`isRegistered: true`・`deletedAt: null`の条件に加え、`employeeSkills`(skillId・skillカテゴリ名)と`employeeCertifications`(certificationId・資格名)を`select`で取得
5. 取得した社員配列から、社員ごとにスキルid・資格idを重複排除した中間データを作り、`aggregateSkillHolders`/`aggregateCertificationHolders`に渡して集計結果を得る
6. EMPLOYEE役割の場合のみ自分の`organizationUnitId`を追加で1クエリ取得し、集計結果に登場する全社員について`canViewEmployeeResume`で`canView`フラグを計算(`Map<employeeId, boolean>`)。HR_SALES/MANAGERは全員`true`固定
7. `OrganizationUnitRadioFilter`(組織単位選択)+`SkillHolderList`を2セット(スキル用・資格用)描画

## 検証方法

- `npm run verify`
- Playwright(開発用ログイン)で以下を確認:
  1. トップ→スキルマップタイルでREF008に遷移でき(全ロール共通)、初期表示で全社集計が表示される
  2. 組織単位(事業部/部署/Gr)を選択すると、配下含めて集計対象が絞り込まれる
  3. スキル別集計がカテゴリ別にグルーピングされて表示され、資格別集計はグルーピングなしで表示される。退職者(退職一郎)が集計に含まれないこと
  4. 集計項目をクリックすると保有者名が展開表示される。人事・営業/管理職でログインした場合、全員の氏名がREF003へのリンクになっている
  5. 一般社員でログインした場合も保有者名は全員分表示されるが、閲覧範囲外の社員名はリンクにならない(テキストのみ)。閲覧範囲内の社員名はREF003へのリンクとして機能する
  6. リンクからREF003に遷移でき、既存の閲覧範囲ガード(直接URL・REF003側)が引き続き機能する(回帰なし)

### 検証結果

`npm run verify`(lint・tsc・vitest 266件)が通過。Playwrightで以下をすべて確認した。

1. トップ→スキルマップタイル(「準備中」解除)→REF008に遷移でき、初期表示(全社)で登録済み・在職中の5名が集計対象になった

6. 検証用に変更したデータ(山田五郎のJavaスキル登録・所属組織)はすべて元の状態(第一Gr所属・スキル未登録)に復元し、後片付けした

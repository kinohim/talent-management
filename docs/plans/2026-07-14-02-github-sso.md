# GitHub OAuth 実SSOログイン 実装計画

## Context

これまで認証は開発用ログイン(社員ID + Credentialsプロバイダ)のみで、実SSOは `lib/auth.ts` のTODOのまま未実装だった。仕様(docs/screens.md AUTH001 / docs/decisions.md「認証」)ではAzure AD／Google／GitHubの3プロバイダ・メールアドレス事前登録照合・初回プロバイダ固定が確定している。本計画ではこのうちGitHubプロバイダによる実SSOログインを実装する。

確定スコープ(ユーザー確認済み):

- 実装するのは**GitHubプロバイダのみ**。Azure AD／Googleはログイン画面に非活性ボタン(「準備中」表示)として配置
- **開発用ログインは残す**(実運用開始前のため)。UI上「開発用ログイン」であることを明記
- OAuthアプリのクレデンシャルは未取得のため、実装+単体テストまで。実プロバイダでのE2E確認はクレデンシャル取得後に別途行う

## 技術判断(@auth/core 0.41.2 のソースで検証済み)

1. **signInコールバックはアダプタのcreateUser/linkAccountより前に実行される**。文字列を返すとそのURLへリダイレクトされる。これで`employeeId`必須のUserが未登録メールで自動作成される事故を確実に防ぎ、OAuthフローのエラーを`/login?error=<code>`で画面に伝えられる
2. **初回ログイン時のaccounts紐付けは`allowDangerousEmailAccountLinking: true`による自動リンク**。signInコールバックで (a)確認済みメール限定 (b)事前登録照合 (c)プロバイダ不一致チェックを先に実施するため安全。この照合とセットで維持することが前提
3. **標準GitHubプロバイダは`/user/emails`の`verified`フラグを確認しない** → `userinfo.request`をオーバーライドし「verifiedかつprimary」のメールだけを採用。該当なしは「未登録」エラーに丸める(なりすまし防止)
4. **セッション戦略は現行の`useDevLogin ? "jwt" : "database"`を維持**。OAuthはjwt戦略でもアダプタ経由のユーザー/アカウント永続化が行われる
5. **クレデンシャル未設定環境対策**: `AUTH_GITHUB_ID`/`AUTH_GITHUB_SECRET`の有無(`lib/env.ts`の`isGitHubSsoEnabled`)でプロバイダ登録とボタン活性を切り替える

## 実装ファイル

### 新規

- `lib/sso-login.ts`: テスト可能な照合ロジック(lib/dev-login.tsのパターン踏襲)
  - `pickVerifiedPrimaryEmail(emails)`: GitHub `/user/emails`レスポンスからverifiedかつprimaryのメールを返す(なければnull)
  - `resolveSsoLogin({ email, provider })`: 事前登録照合。判別可能ユニオン`{ok: true} | {ok: false, errorCode}`を返す。emailなし/該当ユーザーなし→`not-registered`、退職済み→`retired`、別プロバイダで紐付き済み→`provider-mismatch`(初回プロバイダ固定)、初回or同一プロバイダ→ok。メールは小文字化して照合
- `lib/sso-login.test.ts`: 上記のテスト(Prismaモック、10ケース)
- `components/auth/SsoLoginButtons.tsx`: SSOボタン群(Server Component)。Microsoft/Googleは非活性「(準備中)」、GitHubは`gitHubEnabled`に応じて活性(form action)/非活性「(未設定)」

### 変更

- `lib/env.ts`: `isGitHubSsoEnabled`を追加
- `lib/auth.ts`:
  - GitHubプロバイダ追加(`allowDangerousEmailAccountLinking` + `userinfo`オーバーライド。userinfoは部分マージでなく全体置換のため`url`併記)
  - `callbacks.signIn`追加: 開発用ログインは素通し、OAuth/OIDCは`resolveSsoLogin`で照合し、エラー時は`/login?error=<code>`文字列を返してリダイレクト
  - `events.signIn`拡張: HR_SALESのSSOログイン時、`employee.name`が未設定ならSSOの表示名で補完(screens.md AUTH001「初回ログイン後の遷移」。設定済みの名前は上書きしない)
  - 冒頭の実SSO TODOコメントを残TODO(Azure AD/Google)に更新
- `lib/auth-errors.ts`: `messageForLoginErrorCode(code)`を追加(OAuthフローのクエリcode→日本語文言)。throwされることのなくなった`ProviderMismatchError`クラスは削除
- `app/login/actions.ts`: `githubLoginAction`を追加(`signIn("github", { redirectTo: "/" })`。エラーはリダイレクトで戻るためcatch不要)
- `app/login/page.tsx`: `searchParams.error`を`messageForLoginErrorCode`で日本語表示(`role="alert"`)。SSOボタン群を常時表示し、`useDevLogin`時のみ区切り付き「開発用ログイン」セクションを併設(「SSOログインは準備中です」分岐は削除)
- `.env.example`: `AUTH_SECRET`/`AUTH_GITHUB_ID`/`AUTH_GITHUB_SECRET`とGitHub OAuthアプリ登録手順(コールバックURL: `/api/auth/callback/github`)を追記
- `docs/decisions.md`「認証」: 実装済み範囲・verifiedメール限定・自動リンクの前提・エラーのリダイレクト伝達方式を反映
- `docs/screens.md` AUTH001: 非活性配置・開発用ログイン併設の現状を注記

## 検証

- `npm run verify`: 全項目パス(テスト290件)
- 開発サーバー + Playwrightでの手動確認(クレデンシャル未設定状態):
  - `/login`でMicrosoft/Google「(準備中)」・GitHub「(未設定)」が非活性、「開発用ログイン」セクションが区分表示されること
  - `/login?error=not-registered|retired|provider-mismatch`で各日本語文言、未知codeでデフォルト文言が表示されること
  - 開発用ログインの回帰(未登録IDでエラー表示、登録済みIDでログイン成功)
- 残タスク(スコープ外): GitHub OAuthアプリ登録後に実クレデンシャルでのE2E確認(初回ログインでのaccounts自動紐付け、プロバイダ不一致エラー)。Azure AD/Googleプロバイダの実装

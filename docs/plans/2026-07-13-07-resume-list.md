# 実行計画: 経歴書一覧(REF002)

## Context

REF001(トップ)の「経歴書一覧」タイルは`href`未設定だった。アカウント管理系(REF007/EDT006/EDT007)に続き、REF002(経歴書一覧)を実装する。REF002は全社員の経歴書を検索・一覧表示する画面で、REF003(経歴書詳細)の閲覧範囲判定ロジック(`lib/organization-unit.ts`の`canViewEmployeeResume`/`isWithinResumeViewScope`)を実装時から「REF002でも再利用する」前提で設計されていた(`docs/plans/2026-07-10-03-resume-detail.md`参照)。

また、REF003実装時に「人事・営業/管理職がREF002から遷移した場合の『一覧に戻る』表示」はREF002実装時に対応する設計判断済みの積み残しであり、今回あわせて実装した。

## 画面仕様(docs/screens.md REF002)

| 項目 | 種別 | 必須 | 内容 | 備考 |
|---|---|---|---|---|
| 氏名・カナ | テキスト | 任意 | 50文字以内 | 部分一致検索(氏名/カナのどちらかに一致) |
| 所属組織 | 階層選択 | 任意 | 事業部／部署／Grから選択 | 上位を選ぶと配下も含めて検索。一覧表示は最下層のみ |
| 経験年数(下限〜上限) | 数値 | 任意 | 0〜99の整数 | 下限≦上限 |
| スキル条件 | 複数選択+サジェスト | 任意 | スキルマスタから複数選択 | AND/OR切替(トグル) |
| 取得資格条件 | 複数選択+サジェスト | 任意 | 資格マスタから複数選択 | AND/OR切替(トグル) |
| 検索結果一覧 | 表示 | - | 氏名／所属組織／経験年数／主なスキル | 「詳細」でREF003へ。退職者はデフォルト除外、フィルタで表示切替可 |

一般社員は閲覧範囲内(自分を含む)のみ検索対象。所属組織フィルタの選択肢も自身の閲覧範囲内に限定する。基本条件・スキル条件・資格条件はそれぞれ折りたたみ可能なセクション(`<details>`で実装、追加JS不要)。

## 閲覧範囲の実装方針(既存ロジックを拡張)

既存の`isWithinResumeViewScope`/`canViewEmployeeResume`(`lib/organization-unit.ts`)は「1対1」の閲覧可否判定用。REF002では検索対象の母集団を1回のDBクエリで絞り込む必要があるため、新規に`resolveResumeViewScopeUnitIds(units, viewerOrganizationUnitId): Set<number>`を追加した。

- 事業部直下所属 → 事業部全体を対象(ルールb)
- 部署以下(部署/Gr)所属 → 遡って到達する部署ノードとその配下(Gr)に加え、**事業部直下所属者(ルールbで相手が事業部直下なら双方向で許可されるため)も対象に含める**
- 未所属 → 空集合(ルールc。本人はemployeeIdでの別枠一致により引き続き検索可能)

一般社員でのクエリ組み立ては`organizationUnitId: { in: [...scopeIds] } OR employeeId: 自分`を必須条件に加える。所属組織フィルタで特定の組織を選んだ場合も、選択id(配下展開後)と`scopeIds`の積集合をとる(URLクエリはユーザー操作可能なため、閲覧範囲外を指定されてもサーバー側で必ずクランプする)。人事・営業/管理職は無条件で全社員が対象(所属組織フィルタはただの絞り込み)。

## 設計判断

1. **「主なスキル」列**: 同一スキルを複数バージョンで登録している場合(例: Java 8とJava 11)に名前が重複するため、スキル名を`Set`で一意化してから最大3件+超過時は「…」を付けて表示する。
2. **AND/OR初期値**: 未指定時はOR(より広い母集団を初期表示する方が一般的なUXのため)。
3. **並び順**: 氏名昇順(`orderBy: { name: "asc" }`)。登録済み社員のみが対象のため`name`は必ず値を持つ。
4. **退職者含めるトグルは全ロール共通**: 仕様にロール限定の記述が無いため、一般社員にも表示する(閲覧範囲内の退職者を参照したいケースを許容)。
5. **REF003の「一覧に戻る」リンク**: グローバルパンくず(`/resumes/[id]`の親は「マイページ」固定)は本人プレビュー導線と両立できないため変更せず、`!isSelf`時のみ表示するページ固有のリンクとして追加した。旧仕様コメントでは「人事・営業/管理職から遷移」とあったが、REF002実装後は一般社員も閲覧範囲内の他者を見られるため、条件は役割問わず`!isSelf`とした。

## バグ修正1: 一般社員の所属組織フィルタが空になる

実装直後の動作確認で、一般社員(鈴木花子)でREF002を開くと所属組織フィルタが「組織単位が登録されていません」と表示される不具合を発見した。

原因は`buildOrganizationUnitTree`(`lib/organization-unit-tree.ts`)が`parentId === null`のユニットのみをツリーの根として探索する実装だったこと。REF002では一般社員の閲覧範囲(`scopeUnitIds`)でユニットを絞り込んでから渡すため、根になるはずの部署ノードの親(事業部)がリストに含まれず、かつ部署ノード自身の`parentId`はnullではない。そのため`build(null)`が何も見つけられず空のツリーになっていた。

`buildOrganizationUnitTree`に第2引数`rootIds?: number[] | null`を追加し、指定時はそのidの集合を根として探索を開始するようにした。呼び出し側(`app/(authenticated)/resumes/page.tsx`)では、絞り込み後のユニットのうち「親がスコープ外またはnull」であるものを根として計算して渡す。この判定は事業部/部署のどちらが根になるケースにも汎用的に対応する。

## バグ修正2: REF002の検索結果とREF003の直接アクセス判定の不整合(ルールb未対応)

一般社員での動作確認中、閲覧範囲判定の「事業部直下所属者」ケースで不整合を発見した。部署所属の一般社員(鈴木花子、開発部A/第一Gr)から見て、同じ事業部(システム事業部)に事業部直下所属の社員(000006)がいる場合、REF003への直接URLアクセスは`isWithinResumeViewScope`のルール(b)により許可される(実際に確認済み)一方、REF002の検索結果には表示されなかった。

原因は`resolveResumeViewScopeUnitIds`が、部署以下所属のviewerに対して単純に「部署の配下(collectDescendantIds)」のみを返しており、ルール(b)(「相手が事業部直下なら同一事業部内は双方向で許可」)を考慮していなかったこと。`isWithinResumeViewScope`の1対1判定とロジックが乖離していた。

`resolveResumeViewScopeUnitIds`を修正し、viewerが部署以下所属の場合は「部署の配下」に加えて事業部id自体(事業部直下所属者の`organizationUnitId`と一致するid)も対象に含めるようにした。`lib/organization-unit.test.ts`の該当テストも新しい期待値に更新した。

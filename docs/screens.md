# 画面一覧・詳細仕様

全18画面と共通部品1。画面IDはURLセグメント由来のkebab-caseで、URLと1対1に対応させる（一部は意味を保った略称。例: `/master/organization-units`→`master-org-units`）。mypage内のセクション編集（基本情報／経歴概要・自己PR／スキル／資格）と[実績]タブは独立した画面IDを持たず、mypageの節に含めて記述する。

種類別に記載。各画面の関連テーブルは `schema.md` を参照。全画面共通でパンくずリストによる画面間移動に対応する（loginを除く）。**ヘッダとパンくずリストは画面上部に固定し、スクロールしても常に表示する。**

**ヘッダの共通ナビゲーション**: ヘッダには主要画面への導線（私の経歴書／経歴書一覧／ダッシュボード（skill-map）／マスタ管理／アカウント一覧）を常時表示し、ロールに応じて表示を出し分ける（表示条件はhomeの節のタイルと同じロール判定に従う。「私の経歴書」は一般社員・管理職のみ、「マスタ管理」「アカウント一覧」は管理職のみ、それ以外は全ロール表示）。アプリ名表記（「経歴書」）はトップ（home）へのリンクを兼ねる。「マスタ管理」はフォーカス／ホバーでmaster-homeと同じ5画面（部署／スキル／資格／現場ポジション／現場）へのドロップダウンを表示し、各画面へ直接遷移できる（「マスタ管理」自体のクリックはmaster-homeへ遷移する従来動作を維持）。

区分値（`employment_status`等）の表記は `schema.md` のENUM格納値（`active`/`retired`等）に合わせる。

**認証・認可の共通仕様**: 認証必須ページ（login以外の全画面）は、未ログインならloginへリダイレクトする。一般社員／管理職で初回登録未完了（`is_registered=false`）の場合は、どのページにアクセスしてもbasic-info（初回登録）へ誘導する（`lib/auth-routing.ts` の `resolveDestination` による毎リクエストのガード）。管理職専用画面（account-*／master-*）は、ページ側のロール判定とServer Action側の `requireManager`（`lib/auth-guards.ts`）の二段で保護し、権限のないユーザーはトップへリダイレクトする。

**入力フィールドの共通仕様**: テキスト系の入力フィールドには一律でフィールド内右端に×（クリア）を表示し、1クリックで空にできる（サジェスト（datalist）付きの入力も対象）。日付・年月入力には`min`/`max`を必ず付与して**年を4桁までに制限**する（4桁入力後は月へ自動移動。西暦6桁の入力を防ぐ）。日付・年月入力はネイティブのカレンダーインジケータを非表示にし、**×の左に自前のカレンダーアイコン**を置いてクリックでピッカーを開く（×とピッカー起動領域の重なり防止）。検索機能がある画面（resume-list・account-list）には検索ボタンの隣に「クリア」ボタンを置き、全検索フィールドを一括クリアできる（クリアのみで検索は実行しない）。

**ボタンの共通仕様**: ボタンはカーソルをポインタにし、マウスオーバーで背景色が変わるホバースタイルを持たせる（押せることが視覚的に分かるようにする）。

# 認証

## login ログイン

SSO（Azure AD／Google／GitHub）でログインする。ID/パスワードによるログインは提供しない。実装済みプロバイダはGitHubのみで、Azure AD／Googleのボタンは実装まで非活性（「準備中」表示）で配置する。実運用開始までは開発用ログイン（社員IDログイン）をSSOボタンと区分したうえで併設する（docs/decisions.md「認証」参照）。開発用ログインの説明文は「開発・動作確認専用の仮ログインです。」「社員IDのみでログインできます。」の2行で表示する。

ログイン済みのユーザーが/loginへアクセスした場合はトップ（/）へリダイレクトする。

**ボタンの並び順と配色**: 準備中（非活性）のボタンを先頭に配置する（Microsoft→Google→GitHubの順。GitHubが未設定の環境ではGitHubも準備中扱い）。各ボタンは活性・非活性を問わずプロバイダのイメージカラーで配色する（Microsoft: #0078D4ブルー、Google: 白地+#4285F4、GitHub: #24292Fブラック）。非活性ボタンには時計アイコン＋「(準備中)」表記を付け、非活性であることを明示する。

| 項目 | 種別 | 必須 | 内容 | 備考 |
|---|---|---|---|---|
| Microsoftアカウントでログイン | ボタン | - | Azure AD (OIDC) へリダイレクト | |
| Googleアカウントでログイン | ボタン | - | Google OAuthへリダイレクト | |
| GitHubアカウントでログイン | ボタン | - | GitHub OAuthへリダイレクト | 個人アカウントで会社メールと紐付いていない場合は未登録エラーになる |

**エラー表示（条件付き）**

- 未登録（コード`not-registered`）: 認証プロバイダから取得した確認済みメールアドレスが`users.email`に該当レコードなし →「このメールアドレスは登録されていません。管理者に新規登録を依頼してください。」
- 退職済み（コード`retired`）: 該当employeeの`employment_status = retired` →「このアカウントは無効化されています。心当たりがない場合は管理者にお問い合わせください。」開発用ログインでも同じ判定を行う
- プロバイダ不一致（コード`provider-mismatch`）: 紐付き済みの`accounts.provider`と異なるプロバイダで認証された →「このアカウントは別のログイン方法で登録されています。初回に使用したログイン方法をお使いください。」ログインは実行しない
- 上記以外（Auth.js標準エラー・未知のコード）は既定文言「ログインできませんでした。時間をおいて再度お試しください。」に丸める

エラーの伝え方はプロバイダ種別で異なる: 開発用ログイン（Credentials）はServer Actionの戻り値としてフォームに表示、SSO（OAuth）はフローの途中でthrowを画面に返せないため`/login?error=<code>`へのリダイレクトで伝え、loginがコードを上記文言に変換して表示する（`lib/auth-errors.ts`）。

**ログイン後の遷移**: ログイン成立後は常にトップ（/）へリダイレクトする。一般社員／管理職で`is_registered=false`の場合は、認証必須ページ共通のガード（冒頭「認証・認可の共通仕様」参照）によりbasic-info（初回登録）へ誘導される。**人事・営業は経歴書を作成しないため、ログイン成立時に`is_registered`を自動でTRUE、`employee.name`が未設定ならSSOの表示名を設定し**（account-listの状態表示が「初回未登録」のまま残らないようにするため）、basic-infoを経ずhome（トップ）を利用する。氏名の自動補完は**SSOログイン時のみ**行う（開発用ログインには対応する情報源がないため対象外）。

**プロバイダの扱い**: 1アカウントにつき、初回ログインで使用したプロバイダ（紐付き済み`accounts.provider`）に固定される。2回目以降に別プロバイダで認証された場合は上記「プロバイダ不一致」エラーとする（切替機能は提供しない）。

## resume-list 経歴書一覧

全ロールが全社員の経歴書を検索・一覧表示する（一般社員もロールによる検索対象の限定はなく、所属組織フィルタも全組織を選択できる）。
ただし詳細（resume-detail）への導線は、一般社員は**閲覧範囲内**の社員（自分を含む。下記判定ルール参照）に限定する。範囲外の社員の行には「詳細」リンクを表示しない（リンク非表示。URL直接指定でもresume-detail側が同じ判定で拒否しトップへリダイレクトする）。「PDF」ボタン（pdf-previewへの導線）は**人事・営業／管理職には全行に、一般社員には自分自身の行にのみ表示**する（一般社員のPDF出力は自身の経歴書のみ）。
**検索対象は登録済み（`is_registered=true`）の社員のみ**。管理職が仮登録しただけで本人が初回登録を済ませていない社員は、ロールを問わず一覧・詳細のどちらにも表示されない（フィルタでの表示切替も不可）。
検索条件は開閉可能なカードに配置する。ヘッダは「▼ 検索条件」（開閉アイコンを見出しの左に置く）とし、その**すぐ右隣**に「**検索後に閉じる**」ON/OFFトグルを表示する（**デフォルトON**）。ONのとき検索実行でカードを自動的に閉じ、OFFなら開いたままにする。検索実行中（結果の再取得中）は**画面全体にローディングオーバーレイ**を表示する。開閉状態・トグルはユーザーの操作をブラウザタブ単位で記憶し、記憶がなければ「開」とする。項目順は**氏名カナ→経験年数→所属組織→スキル条件→取得資格条件→携わったプロジェクト（現場）→退職者を含める→検索／クリアボタン**。テキスト系フィールドは幅を抑えたコンパクトな表示とする。

**閲覧範囲の判定ルール（一覧の「詳細」導線の出し分けと、resume-detailの閲覧権限が本ルールに従う）**: 一般社員が他の社員の経歴書（詳細）を閲覧できるかは、双方の所属組織単位（`employee.organization_unit_id`）から親を遡って判定する。（a）双方とも部署以下（部署またはGr）に所属している場合は、遡って到達する**部署（`unit_level=department`）の一致**で判定する（Gr所属なら親の部署）。（b）**どちらか一方が事業部直下所属の場合は、遡って到達する事業部（`unit_level=division`）の一致**で判定する（事業部直下の社員は同一事業部の全社員と相互に閲覧できる）。（c）未所属（NULL）の社員は対象外で、見る側・見られる側のどちらにもならない（本人の経歴書のみ閲覧可／人事・営業・管理職からは従来どおり閲覧可）。

| 項目 | 種別 | 必須 | 内容 | 備考 |
|---|---|---|---|---|
| 氏名・カナ | テキスト | 任意 | 50文字以内 | 氏名・カナのOR部分一致検索（大文字小文字無視） |
| 経験年数（下限〜上限） | 数値 | 任意 | 0〜99の整数 | 下限≦上限（逆転入力は自動で入れ替え）。年入力を月数に換算して判定し、上限は端数月を含む（上限N年＝N年11か月まで該当） |
| 所属組織 | 階層選択（カスケード） | 任意 | 初期表示は事業部のみ。チェックすると配下（部署→Gr）が展開表示される | 未選択の親には配下件数（「部署3件 ▸」等）と「上位の組織を選択すると配下の組織で絞り込めます」の説明を表示。親と子の両方を選択した場合は最深の選択配下のみが検索対象。一覧表示は最下層のみ |
| スキル条件 | 複数選択＋サジェスト | 任意 | スキルマスタから複数選択 | AND/ORをラジオで選択（「いずれか含む(OR)」／「すべて含む(AND)」） |
| 取得資格条件 | 複数選択＋サジェスト | 任意 | 資格マスタから複数選択 | AND/ORをラジオで選択（同上） |
| 携わったプロジェクト（現場） | 単一選択（セレクトボックス） | 任意 | 現場マスタから1件のみ選択（選ぶだけで検索条件になる。確定ボタンなし） | 過去〜現在のいずれかのプロジェクトの現場が一致すれば該当 |
| 退職者を含める | チェックボックス | 任意 | デフォルトOFF（退職者は検索対象外） | ONにすると退職者（`employment_status=retired`）も検索対象に含める |
| 検索結果一覧 | 表示 | - | 氏名／所属組織／経験年数／主なスキル／主な資格 | 「詳細」でresume-detailへ（一般社員は閲覧範囲内の社員の行のみ「詳細」を表示し、範囲外はリンクなし）。「PDF」でpdf-preview（`/resumes/[employeeId]/pdf-preview?from=list`）へ（人事・営業／管理職は全行、一般社員は自分自身の行のみ表示。遷移元を`?from=list`で引き継ぎ、戻り・パンくずの上位は経歴書一覧になる）。主なスキル・主な資格は重複除去後の上位3件＋超過は「…」で省略。経験年数は年換算（月数を12で割った整数部）、未計算は「-」、未所属は「未所属」 |

**表示件数とページ送り**: 一覧の右上（検索フィールドの下）に全件数・ページ番号（前へ／次へ）・表示件数セレクト（10件／30件（既定）／50件／100件）を表示する。検索条件の変更時は1ページ目に戻る（表示件数の設定は引き継ぐ）。件数の減少等でページ番号が実在範囲を超えた場合は最終ページに丸める（空ページは表示しない）。

**絞り込みの保持**: 詳細（resume-detail）へ遷移して「経歴書一覧に戻る」（パンくず含む）で戻った場合、直前の絞り込み・ソート・ページを復元する（ブラウザタブ単位で記憶）。

**一覧ヘッダのソートと絞込**: 列ヘッダのラベルクリックで昇順→降順→解除をトグルする（ソート可能列: 氏名（カナ順）／所属組織／経験年数。主なスキル・主な資格はソート不可）。既定の並び順は氏名（カナ）昇順で、全ソートに社員ID昇順のタイブレークを付ける。各列のフィルタアイコンからポップアップで列単位の絞込ができる（氏名はテキスト部分一致、所属組織は値の選択式（「未所属」含む）、経験年数は範囲指定、**主なスキル・主な資格は検索フォームと同仕様のマスタ複数選択＋サジェスト＋AND/OR選択**）。列フィルタは検索フォームの条件とAND合成される。**列フィルタの所属組織は選択した組織単位の配下も含めて判定する**（検索カードのカスケード選択と同じ配下展開。例: SI事業部を選択すると、配下の金融サービス部などの部署・Grに所属する社員も該当する）。ポップアップは一覧の行数が少なくても見切れないよう画面基準で表示する。

## mypage 私の経歴書

本人が自分の経歴書を1画面で閲覧・編集する画面。閲覧表示はresume-detail（経歴書詳細）と同じ表示コンポーネントを使い、プレビューを兼ねる。画面上部に2つのタブを持つ。基本情報／経歴概要・自己PR／スキル／資格の編集は本画面のセクション編集、プロジェクト経歴一覧は[実績]タブが担い、いずれも独立した画面IDを持たない（各セクションの入力仕様は下記）。

### [表紙]タブ（デフォルト）

プロジェクト経歴以外の全項目を、セクション単位で表示する。各セクションは通常は閲覧表示で、「編集」ボタンでそのセクションのみ編集フォームに切り替わり、保存すると閲覧表示に戻る（「キャンセル」で入力を破棄して閲覧表示へ）。保存時に画面遷移はしない。**セクション見出しは編集/非編集を問わず同じ位置に同じ文言で表示する**（「◯◯の編集」への切替はしない）。

編集モード中はセクション見出し行の右側に**[保存][キャンセル]を横並び**で表示し、フォーム下部には保存ボタンを置かない。セクション保存は画面遷移せず、該当セクションの編集モードを解除して最新の閲覧表示に戻る。

|---|---|
| 基本情報（最終学歴含む） | basic-info参照（初回登録の単独画面と共通の入力仕様。社員ID・メールアドレスの表示のみ初回登録画面限定） |
| 経歴概要・自己PR | 下記「経歴概要・自己PRセクション」参照（AI生成含む） |

| 資格 | 下記「資格セクション」参照 |

### 経歴概要・自己PRセクション

| 項目 | 種別 | 必須 | 内容 |
|---|---|---|---|
| 経歴概要 | テキストエリア | 任意 | 1000文字以内、文字数カウンター表示 |
| 自己PR | テキストエリア | 任意 | 1000文字以内、文字数カウンター表示 |

テキストエリアはリサイズ可能とする。各項目は「登録用フォーム｜←ボタン｜AI生成パネル」の横並びで、**登録用フォームとAI生成パネルは同じ枠スタイル・同じ高さに揃える**。

**AI生成**: 登録済みのデータ（プロジェクト経歴・スキル・資格・経験月数）から経歴概要・自己PRの文章案を生成する。人事・営業は本機能の対象外（API側でも403で拒否する。文言「この機能を利用する権限がありません。」）。

- AI生成パネルには「Geminiで生成」（Gemini API）と「Claudeで生成」（Claude API）の2つのボタンを並べて表示し、利用者が生成に使うAIを選択できる。生成中はクリックした側のボタンのみ「生成中...」表示にする
- **生成結果は保存されない**。パネルに「この内容は保存されません。反映するには←ボタンで入力欄へコピーしてください」と表示し、利用者が「←」ボタンで登録用フォームへコピーして修正・保存する。生成結果は入力上限に合わせて先頭1000文字に切り詰める
- **上書き確認**: 生成結果パネルに既に文章がある状態で再度生成すると「出力フォームの内容を置き換えます。よろしいですか？」の確認を表示する。「以後このセッションでは確認しない」チェックでスキップでき、記憶はブラウザセッション内のみ有効（経歴概要・自己PR共通）
- **データ不足時の案内**: プロジェクト経歴・スキル・資格のいずれかが未登録でも生成は実行し、生成後に未登録のデータ種別を挙げて「登録するとより精度の高い文章を生成できます」の案内を表示する
- **エラー表示**: 無料枠上限・レート制限（「AI生成の無料利用枠の上限に達しました。…」「AI生成が混み合っています。…」）、クレジット残高不足（「AI生成機能のクレジット残高が不足しています。管理者にお問い合わせください。」）、APIキー未設定（「AI生成機能が利用できません(サーバー設定エラー)。…」）、生成結果が空・AIが生成を拒否（「AIが文章を生成できませんでした。時間をおいて再度お試しください。」）、その他の失敗（「AI生成に失敗しました。時間をおいて再度お試しください。」）、通信エラー（「AI生成に失敗しました。通信環境をご確認のうえ再度お試しください。」）を区別してパネル内に表示する

## pdf-preview PDF出力プレビュー

経歴書をPDFとして出力（ダウンロード）する直前に、レイアウトとマスク内容を確認する画面。経歴書一覧の「PDF」ボタン・resume-detailの「PDF出力」ボタン・mypageの「PDF出力」ボタンから遷移する。出力形式はPDFのみで、Excel出力は提供しない。表示セクション（基本情報・最終学歴・経歴概要・自己PR・スキル一覧・資格一覧・プロジェクト経歴）の表示仕様はresume-detail／mypageの閲覧表示と共通。

URLは遷移元ごとに分かれる（`/resumes/[employeeId]/pdf-preview`＝経歴書一覧・resume-detail発、`/mypage/pdf-preview`＝mypage発・本人のみ）。**PDF出力（ダウンロード）の権限は本人・人事・営業・管理職のみ**: 一般社員が出力できるのは自身の経歴書だけで、閲覧範囲内の他人でも本画面は開けない（URL直接指定でもトップへリダイレクト。他人の閲覧はresume-detailで行う）。対象は登録済み（`is_registered=true`）の社員のみ（未登録はURL直接指定でもトップへリダイレクト）。

PDF生成はブラウザ印刷ベース（`window.print()`＋印刷用CSS）とする。ボタンは**「ダウンロード」1つに統一**（印刷ボタンは置かない）し、押下でブラウザの印刷ダイアログを開いて送信先「PDFに保存」で保存する。ボタンは画面右上に配置し、ボタンの右端を経歴書シート（A4幅）の右端に揃える。**画面名（見出し）の右に注意書きを表示する**:「マスクした項目（グレー表示）は出力されず、レイアウトは出力時に詰め直されます。ダウンロード押下後の印刷ダイアログのプレビューで確認し、送信先「PDFに保存」で保存してください。」。プレビューはA4サイズのシート表示で、印刷結果と同一のレイアウト・配色（白背景・黒文字。ダークモードでも白紙面に固定）とする。

出力項目の制御（社外提出用）: 切り替えコントロールは画面上部のパネルではなく**対象項目のすぐ右横**に置く（コントロール類は印刷時に出力されない）。

## skill-map スキルマップ／組織ダッシュボード

# マスタ管理系

**共通仕様（同名の論理削除済み行の復活）**: システム全体ユニークな名称（スキル名・資格名・現場ポジション名・現場名）およびカテゴリ名（スキル／資格）は、DBのユニーク制約が論理削除済み行にも及ぶ。このため各マスタ画面での新規登録時、同名の論理削除済み行が存在する場合は新規行を作らず**削除済み行を復活させる**（`deleted_at`等をクリアし、カテゴリ・認定団体・主管部署等の入力値は今回の値で上書きする。idと作成日時は元の行のまま）。現役の同名行との重複は従来どおりエラーで弾く（docs/decisions.md「スキル・資格・マスタ全般」参照）。

## master-org-units 部署マスタ管理

組織単位（事業部＞部署＞Gr）の階層を管理する。初期表示は事業部のみで、行頭の開閉ボタン（▶/▼）で配下を展開する。閉じている行には配下の件数（「部署3件」等）を表示し、下位組織の存在が分かるようにする。各行は「名称は左、[編集][配下に追加][削除]ボタンは右端」に統一配置する。

| 項目 | 種別 | 必須 | 内容 | 備考 |
|---|---|---|---|---|
| 事業部を追加 | テキスト＋ボタン（画面最上部に常時表示・コンパクトな1行形） | 必須 | 事業部名100文字以内。新規事業部（unit_level=division, parent_id=NULL）を作成 | 「配下に追加」のフォームと同じサイズ感 |
| 配下に追加 | ボタン（一覧内） | - | その行の直下階層として新規組織単位を追加（追加後は自動で展開） | Gr（最下層）の行には表示しない |
| 一覧（階層表示）・編集・削除 | 表示＋ボタン | - | 事業部＞部署＞Grの開閉式インデント表示 | 配下の子組織・所属社員・現場マスタの主管部署としての参照のいずれかが存在する行は削除不可（「使用中のため削除できません」） |

**同名チェック**: 組織単位名は同一親の配下内で一意（別の親の配下なら同名可）。追加・名称変更時に重複する場合は「同じ階層に同名の組織単位が既に存在します。」を表示する。

初期データ（事業部・部署・Grの初期構成）は本番リリースには含めず、リリース後に運用担当者（管理職）が本画面から登録する。開発・デモ用シード（`prisma/seed.ts`）が動作確認用の組織構成を投入するのは本ルールの対象外。

## master-sites 現場マスタ管理

| 項目 | 種別 | 必須 | 内容 |
|---|---|---|---|
| 現場を追加 | テキスト＋主管部署選択＋ボタン（画面最上部に常時表示・コンパクトな1行形） | 現場名のみ必須 | 現場名100文字以内、システム全体でユニーク |

## confirm-dialog 削除確認モーダル

一覧・編集画面から削除操作を行う際に表示する確認ダイアログ。

| 項目 | 種別 | 内容 |
|---|---|---|
| 確認メッセージ | 表示 | 呼び出し元が必ず指定する（例:「「◯◯」を削除してもよろしいですか？」。account-editの退職処理・現職復帰は専用文言を指定） |
| キャンセルボタン | ボタン | モーダルを閉じる（ラベル既定値「キャンセル」） |
| 実行ボタン | ボタン（危険操作） | ラベル既定値は「削除する」（呼び出し元で変更可）。デフォルトは論理削除（deleted_atをセット）。呼び出し元により動作が異なる（account-edit参照） |

**マスタ削除時の参照チェック**: master-skills〜master-sitesの各マスタ画面からの削除では、実行前に対象マスタへの参照有無をチェックし、参照が存在する場合は削除を実行せずエラー「使用中のため削除できません」を表示する（schema.md 一般ルール参照）。

呼び出し元: project-form（プロジェクト経歴の削除）、master-skills／master-certifications／master-project-roles／master-org-units／master-sites（マスタ一覧の削除アクション全般）、account-edit（退職処理・現職復帰・社員ID/メールアドレスの変更確認）。account-editの退職処理は本モーダルを使うが、動作は論理削除ではなく`employment_status`の更新である点に注意。

※mypageのスキル・資格セクションの明細行の削除は、保存前の画面内操作のため本モーダルは使わない（保存時に確定）。

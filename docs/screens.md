# 画面一覧・詳細仕様

全22画面。種類別に記載。各画面の関連テーブルは `schema.md` を参照。全画面共通でパンくずリストによる画面間移動に対応する（AUTH001 ログインを除く）。**ヘッダとパンくずリストは画面上部に固定し、スクロールしても常に表示する。**

**入力フィールドの共通仕様**: テキスト系の入力フィールドには一律でフィールド内右端に×（クリア）を表示し、1クリックで空にできる（サジェスト（datalist）付きの入力も対象）。日付・年月入力には`min`/`max`を必ず付与して**年を4桁までに制限**する（4桁入力後は月へ自動移動。西暦6桁の入力を防ぐ）。日付・年月入力はネイティブのカレンダーインジケータを非表示にし、**×の左に自前のカレンダーアイコン**を置いてクリックでピッカーを開く（×とピッカー起動領域の重なり防止）。検索機能がある画面（REF002・REF007）には検索ボタンの隣に「クリア」ボタンを置き、全検索フィールドを一括クリアできる（クリアのみで検索は実行しない）。

**ボタンの共通仕様**: ボタンはカーソルをポインタにし、マウスオーバーで背景色が変わるホバースタイルを持たせる（押せることが視覚的に分かるようにする）。

# 認証

## AUTH001 ログイン

SSO（Azure AD／Google／GitHub）でログインする。ID/パスワードによるログインは提供しない。実装済みプロバイダはGitHubのみで、Azure AD／Googleのボタンは実装まで非活性（「準備中」表示）で配置する。実運用開始までは開発用ログイン（社員IDログイン）をSSOボタンと区分したうえで併設する（docs/decisions.md「認証」参照）。開発用ログインの説明文は「開発・動作確認専用の仮ログインです。」「社員IDのみでログインできます。」の2行で表示する。

**ボタンの並び順と配色**: 準備中（非活性）のボタンを先頭に配置する（Microsoft→Google→GitHubの順。GitHubが未設定の環境ではGitHubも準備中扱い）。各ボタンは活性・非活性を問わずプロバイダのイメージカラーで配色する（Microsoft: #0078D4ブルー、Google: 白地+#4285F4、GitHub: #24292Fブラック）。非活性ボタンには時計アイコン＋「(準備中)」表記を付け、非活性であることを明示する。

| 項目 | 種別 | 必須 | 内容 | 備考 |
|---|---|---|---|---|
| Microsoftアカウントでログイン | ボタン | - | Azure AD (OIDC) へリダイレクト | |
| Googleアカウントでログイン | ボタン | - | Google OAuthへリダイレクト | |
| GitHubアカウントでログイン | ボタン | - | GitHub OAuthへリダイレクト | 個人アカウントで会社メールと紐付いていない場合は未登録エラーになる |

**エラー表示（条件付き）**
- 未登録: 認証プロバイダから取得した確認済みメールアドレスが`users.email`に該当レコードなし →「このメールアドレスは登録されていません。管理者に新規登録を依頼してください。」
- 退職済み: 該当employeeの`employment_status = 2（退職）` →「このアカウントは無効化されています。心当たりがない場合は管理者にお問い合わせください。」
- プロバイダ不一致: 紐付き済みの`accounts.provider`と異なるプロバイダで認証された →「このアカウントは別のログイン方法で登録されています。初回に使用したログイン方法をお使いください。」ログインは実行しない

**初回ログイン後の遷移（ロール別）**: 一般社員／管理職は`is_registered=false`ならEDT001（初回登録）へ。**人事・営業は経歴書を作成しないため、EDT001を経ずREF001（トップ）へ直行し、ログイン成立時に`is_registered`を自動でTRUE、`employee.name`にSSOの表示名を設定する**（REF007の状態表示が「初回未登録」のまま残らないようにするため）。

**プロバイダの扱い**: 1アカウントにつき、初回ログインで使用したプロバイダ（紐付き済み`accounts.provider`）に固定される。2回目以降に別プロバイダで認証された場合は上記「プロバイダ不一致」エラーとする（切替機能は提供しない）。

## REF002 経歴書一覧

人事・営業／管理職が全社員の経歴書を検索・一覧表示する。
一般社員は**閲覧範囲内**の社員（自分を含む。下記判定ルール参照）のみを検索・一覧表示できる（検索対象がサーバー側で自動的に限定される。所属組織フィルタも自身の閲覧範囲内のみ選択可能）。
検索条件は開閉可能なカードに配置する。ヘッダは「▼ 検索条件」（開閉アイコンを見出しの左に置く）とし、その**すぐ右隣**に「**検索後に閉じる**」ON/OFFトグルを表示する（**デフォルトON**）。ONのとき検索実行でカードを自動的に閉じ、OFFなら開いたままにする。検索実行中（結果の再取得中）は**画面全体にローディングオーバーレイ**を表示する。開閉状態・トグルはユーザーの操作をブラウザタブ単位で記憶し、記憶がなければ「開」とする。項目順は**氏名カナ→経験年数→所属組織→スキル条件→取得資格条件→携わったプロジェクト（現場）→退職者を含める→検索／クリアボタン**。テキスト系フィールドは幅を抑えたコンパクトな表示とする。

**閲覧範囲の判定ルール（REF003の閲覧権限も本ルールに従う）**: 一般社員が他の社員の経歴書を閲覧できるかは、双方の所属組織単位（`employee.organization_unit_id`）から親を遡って判定する。（a）双方とも部署以下（部署またはGr）に所属している場合は、遡って到達する**部署（`unit_level=2`）の一致**で判定する（Gr所属なら親の部署）。（b）**どちらか一方が事業部直下所属の場合は、遡って到達する事業部（`unit_level=1`）の一致**で判定する（事業部直下の社員は同一事業部の全社員と相互に閲覧できる）。（c）未所属（NULL）の社員は対象外で、見る側・見られる側のどちらにもならない（本人の経歴書のみ閲覧可／人事・営業・管理職からは従来どおり閲覧可）。

| 項目 | 種別 | 必須 | 内容 | 備考 |
|---|---|---|---|---|
| 氏名・カナ | テキスト | 任意 | 50文字以内 | 部分一致検索 |
| 所属組織 | 階層選択（カスケード） | 任意 | 初期表示は事業部のみ。チェックすると配下（部署→Gr）が展開表示される | 未選択の親には配下件数（「部署3件 ▸」等）と「上位の組織を選択すると配下の組織で絞り込めます」の説明を表示。親と子の両方を選択した場合は最深の選択配下のみが検索対象。一覧表示は最下層のみ |
| 経験年数（下限〜上限） | 数値 | 任意 | 0〜99の整数 | 下限≦上限 |
| スキル条件 | 複数選択＋サジェスト | 任意 | スキルマスタから複数選択 | AND/ORをラジオで選択（「いずれか含む(OR)」／「すべて含む(AND)」） |
| 取得資格条件 | 複数選択＋サジェスト | 任意 | 資格マスタから複数選択 | AND/ORをラジオで選択（同上） |
| 携わったプロジェクト（現場） | 単一選択（セレクトボックス） | 任意 | 現場マスタから1件のみ選択（選ぶだけで検索条件になる。確定ボタンなし） | 過去〜現在のいずれかのプロジェクトの現場が一致すれば該当 |
| 検索結果一覧 | 表示 | - | 氏名／所属組織／経験年数／主なスキル／主な資格 | 「詳細」でREF003へ。退職者はデフォルトで一覧から除外し、フィルタで表示切替可能とする |

**表示件数とページ送り**: 一覧の右上（検索フィールドの下）に全件数・ページ番号（前へ／次へ）・表示件数セレクト（10件／30件（既定）／50件／100件）を表示する。検索条件の変更時は1ページ目に戻る（表示件数の設定は引き継ぐ）。

**絞り込みの保持**: 詳細（REF003）へ遷移して「経歴書一覧に戻る」（パンくず含む）で戻った場合、直前の絞り込み・ソート・ページを復元する（ブラウザタブ単位で記憶）。

**一覧ヘッダのソートと絞込**: 列ヘッダのラベルクリックで昇順→降順→解除をトグルする（ソート可能列: 氏名（カナ順）／所属組織／経験年数。主なスキル・主な資格はソート不可）。各列のフィルタアイコンからポップアップで列単位の絞込ができる（氏名はテキスト部分一致、所属組織は値の選択式（「未所属」含む）、経験年数は範囲指定、**主なスキル・主な資格は検索フォームと同仕様のマスタ複数選択＋サジェスト＋AND/OR選択**）。列フィルタは検索フォームの条件とAND合成される。ポップアップは一覧の行数が少なくても見切れないよう画面基準で表示する。

## REF004 私の経歴書

本人が自分の経歴書を1画面で閲覧・編集する画面。閲覧表示はREF003（経歴書詳細）と同じ表示コンポーネントを使い、プレビューを兼ねる。画面上部に2つのタブを持つ。

| セクション | 編集単位 | 入力仕様 |

| 経歴概要・自己PR | EDT002と同じ | EDT002参照（AI生成含む） |
| スキル | EDT003と同じ | EDT003参照 |
| 資格 | EDT004と同じ | EDT004参照 |

# 登録・編集系

**画面の提供形態**: EDT001〜EDT004はREF004「私の経歴書」[表紙]タブのセクションとして提供する（セクション内で編集モードに切り替えて保存する。入力仕様は各EDT節のとおり）。編集モード中はセクション見出し行の右側に**[保存][キャンセル]を横並び**で表示し、フォーム下部には保存ボタンを置かない（初回登録の単独画面ではフォーム下部に保存ボタンを置く）。EDT001のみ、初回登録用の単独画面（/register）も持つ。EDT005〜EDT007は従来どおり単独画面。

**保存後の遷移先（共通ルール）**: EDT001〜EDT004のセクション保存は画面遷移せず、該当セクションの編集モードを解除して最新の閲覧表示に戻る。EDT001の**初回登録**（単独画面からの保存）のみREF004へ遷移し、続けてスキル・資格・経歴の入力を促す。EDT005はREF004の[実績]タブへ、EDT006・EDT007はREF007（アカウント一覧）へ戻る。

## MST004 部署マスタ管理

組織単位（事業部＞部署＞Gr）の階層を管理する。初期表示は事業部のみで、行頭の開閉ボタン（▶/▼）で配下を展開する。閉じている行には配下の件数（「部署3件」等）を表示し、下位組織の存在が分かるようにする。各行は「名称は左、[編集][配下に追加][削除]ボタンは右端」に統一配置する。

| 項目 | 種別 | 必須 | 内容 | 備考 |
|---|---|---|---|---|
| 事業部を追加 | テキスト＋ボタン（画面最上部に常時表示・コンパクトな1行形） | 必須 | 事業部名100文字以内。新規事業部（unit_level=1, parent_id=NULL）を作成 | 「配下に追加」のフォームと同じサイズ感 |
| 配下に追加 | ボタン（一覧内） | - | その行の直下階層として新規組織単位を追加（追加後は自動で展開） | Gr（最下層）の行には表示しない |
| 一覧（階層表示）・編集・削除 | 表示＋ボタン | - | 事業部＞部署＞Grの開閉式インデント表示 | 配下または所属社員が存在する行は削除不可（エラー表示） |

**同名チェック**: 組織単位名は同一親の配下内で一意（別の親の配下なら同名可）。追加・名称変更時に重複する場合は「同じ階層に同名の組織単位が既に存在します。」を表示する。

初期データ（事業部・部署・Grの初期構成）はシステム側では投入せず、リリース後に運用担当者（管理職）が本画面から登録する。

## CMN001 削除確認モーダル

一覧・編集画面から削除操作を行う際に表示する確認ダイアログ。

| 項目 | 種別 | 内容 |
|---|---|---|
| 確認メッセージ | 表示 | 呼び出し元から指定可能。デフォルトは「削除してもよろしいですか？」＋対象名（EDT007の退職処理・現職復帰は専用文言を指定） |
| キャンセルボタン | ボタン | モーダルを閉じる |
| 実行ボタン | ボタン（危険操作） | デフォルトは論理削除（deleted_atをセット）。呼び出し元により動作が異なる（EDT007参照） |

**マスタ削除時の参照チェック**: MST001〜005からの削除では、実行前に対象マスタへの参照有無をチェックし、参照が存在する場合は削除を実行せずエラー「使用中のため削除できません」を表示する（schema.md 一般ルール参照）。

呼び出し元: EDT005（プロジェクト経歴の削除）、MST001〜005（マスタ一覧の削除アクション全般）、EDT007（退職処理）。EDT007の退職処理は本モーダルを使うが、動作は論理削除ではなく`employment_status`の更新である点に注意。

※EDT003（スキル）・EDT004（資格）の明細行の削除は、保存前の画面内操作のため本モーダルは使わない（保存時に確定）。
